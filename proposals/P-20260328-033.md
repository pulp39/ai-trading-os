# P-20260328-033
## Dialogue Layer Formalization — Institutional Multi-Agent Dialogue Protocol

**Proposal ID:** P-20260328-033
**Date:** 2026-03-28
**Proposer:** claude_proposer
**Librarian Review:** Accepted（合意済）
**Status:** Draft — Founder承認待ち

---

## 1. Background

AI Trading OS（ATOS）は現在、以下の制度的フローを持つ。

```
Observation → Proposal → Approval → Execution
```

しかし実運用において、ProposerとLibrarianの対話が制度生成において不可欠なプロセスとして機能していることが、本日（2026-03-28）のテストセッションで確認された。

現行モデルではこの対話プロセスが制度的に未定義であり、将来的な解釈の分裂・役割崩壊のリスクを内包している。

---

## 2. Proposal

新たに以下の層を定義し、ATOSの制度的フローに組み込む。

### Dialogue Layer（対話層）

**制度的位置：**

```
Observation
    ↓
Dialogue  ← NEW
    ↓
Proposal
    ↓
Approval
    ↓
Execution
```

---

## 3. Definition

**Dialogue Layer**とは、ProposerとLibrarianの間で行われる、制度生成・構造設計のための非実行対話プロセスである。

- ObservationLayerの出力（観測・解釈）を入力として受け取る
- Proposal Draft候補または設計スケッチを出力する
- READ / WRITE / TRIGGERを発生させない
- 制度確定を行わない（Founder承認が必要）

---

## 4. Properties

| 属性 | 値 |
|------|-----|
| 入力 | Observation Layerの出力（観測・解釈） |
| 出力 | Proposal Draft候補 / 設計スケッチ |
| 状態変更 | なし（READ/WRITE/TRIGGER不発生） |
| 制度確定 | 不可（Founder承認が必要） |
| 実行帰属 | 対象外（非実行層） |

---

## 5. Control Mechanism

Dialogue Layerは以下の制御を持つ。

**5.1 開始条件**
- Founderが対話開始を指示する
- Proposerが開始宣言を行う
- Founderはアクセス先チャネル（スレッド名等）を明示する

**5.2 終了条件（最優先）**
- Founderが終了指示または終了意思表示を行う → 即終了
- ProposerまたはLibrarianが「Founder判断が必要」と宣言する → 即終了

**5.3 合意ベース終了**
- 議論が収束し、双方が結論と認識した場合 → Founder承認待ち状態へ移行

**5.4 デッドロック処理**
- 規定往復数（初期値：10）到達、またはいずれかが停滞と判断した場合 → Founder裁定待ちへ移行

---

## 6. Channel Specification

**現在の実装：** ChatGPTスレッド（Chrome操作機能によるアクセス）

ただし、これは実装依存であり制度要件ではない。将来的にチャネルが変更された場合もDialogue Layerの定義は有効である。

**アクセス制限：**
- ProposerはFounderが名前を指定したチャネルにのみアクセスできる
- 未指定チャネルへの書き込みは禁止
- スレッドへの書き込みはObservation Layerの一部として扱う（制度的確定ではない）

---

## 7. Deviation Policy

本プロトコルにおける逸脱は以下の範囲で許容される。

**逸脱可能：**
- ログ記録の省略または遅延
- 往復回数制限の調整
- 実装手段の変更（例：チャネル変更）

**逸脱不可：**
- 役割分離の破壊
- 無承認WRITE
- Founder権限の代替
- セッション中の制度確定

逸脱が発生した場合は一行で明示することで続行可能とする。

---

## 8. Impact

本Proposalにより以下が実現される。

1. **制度的説明可能性の確立：** ProposerとLibrarianの対話が制度的に位置づけられ、説明可能になる。
2. **Phase 7以降の設計プロセスの安定化：** 対話→設計→実行のフローが明確になる。
3. **AI間協調のスケーラビリティ向上：** Proposer・Librarian以外のAIが将来参加した場合にも、このDialogue Layerが適用される。
4. **Execution Constitutionとの整合：** 三層モデル（Observation/Approval/Execution）を拡張し、対話層を制度内に正式収録する。
