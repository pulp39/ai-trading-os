## 2. 確定した制度的事実

### 2.1 判定状態（readiness_state）の確定

Preview Layer が返す状態は以下の4値に確定・正規化された：

| 状態 | 意味 | 発火条件 |
|---|---|---|
| `READY` | 実行可 | Hard/Soft いずれも違反なし |
| `REPRICE_REQUIRED` | 再評価要 | Soft Limit 違反 |
| `NOT_READY` | 実行不可 | Hard Limit 違反（流動性・スリッページ） |
| `DATA_INVALID` | データ無効 | 判定前条件不成立（データ鮮度 > 60秒 — 判定入力が無効であり Hard Limit 評価以前に棄却） |

- writer 側で `upper()` 正規化済み
- 4状態以外の値は許可しない（制約あり）
- `execution_performed = false` はすべての状態で維持される

### 2.2 判定フロー（確定）

```
if freshness > 60s:
    return DATA_INVALID
elif hard_limit_violated (slippage / ask_qty):
    return NOT_READY
elif soft_limit_violated (dynamic threshold):
    return REPRICE_REQUIRED
else:
    return READY
```

### 2.3 trace_event 構造（確定）

3層イベント構造が制度的分離を反映する：

```
market_observation        （観測）
    ↓
indicator_observation     （解釈）
    ↓
execution_readiness_evaluated  （判断 / Preview）
```

`execution_readiness_evaluated` の主要フィールド：

| フィールド | 内容 |
|---|---|
| `readiness_state` | READY / REPRICE_REQUIRED / NOT_READY / DATA_INVALID |
| `reason` | 判定根拠。フォーマット: `namespace:detail`（例: `overridden:snapshot_stale`、`hard:ask_qty_insufficient`、`soft:slippage_exceeded`） |
| `freshness_seconds` | データ鮮度（秒） |
| `execution_performed` | 常に `false`（Preview Layer 単独では変更不可 — 制度的制約） |

### 2.3.1 reason フォーマット規約

`reason` フィールドは `namespace:detail` の形式に統一する：

| namespace | 意味 | 例 |
|---|---|---|
| `hard` | Hard Limit 違反 | `hard:ask_qty_insufficient`、`hard:slippage_exceeded` |
| `soft` | Soft Limit 違反 | `soft:slippage_exceeded`、`soft:spread_wide` |
| `stale` | 判定前条件不成立（鮮度） | `stale:snapshot_stale`、`stale:freshness_exceeded` |
| `overridden` | 上位条件による上書き | `overridden:snapshot_stale` |
| `ok` | 正常通過 | `ok:all_limits_passed` |

- namespace は小文字固定
- detail は snake_case 固定
- 複合理由は主因の namespace を優先して記録

### 2.4 実証済み検証結果（Phase 7.5-8）

| 状態 | 検証方法 | 結果 |
|---|---|---|
| `READY` | 実市場・連続複数回 | ✔ |
| `REPRICE_REQUIRED` | 動的Soft Limit閾値 | ✔ |
| `NOT_READY` | effective AskQty Hard Limit | ✔ |
| `DATA_INVALID` | 65秒遅延注入 | ✔ |

## 3. 制度的意義

1. **Preview は純粋な判断レイヤー** — Observation・Execution から完全独立
2. **判定は deterministic かつ再現可能** — 同条件で同状態を返す
3. **判定は永続化される** — trace_event として記録・監査可能
4. **Execution 抑止は変更不可の制度制約** — `execution_performed = false` は Preview Layer の出力として固定される。`true` への変更は Execution Gate（P-20260331-002）を通過した場合のみ許可され、Preview Layer 単体では如何なる条件下でも変更できない

---

## 4. Hard Limits との関係

本 Proposal は P-20260330-001（Execution Hard Limits Definition）と整合する。

- Hard Limits は Preview が参照する入力条件として機能する
- Hard Limits 違反 → `NOT_READY`（P-20260330-001 定義通り）
- `DATA_INVALID` は Hard Limit 違反ではなく判定前条件不成立として分類する（データ品質問題であり Limit 評価以前に棄却）
- Hard Limits の変更は Founder Directive として別途管理
