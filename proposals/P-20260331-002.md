## 1. 背景

Preview Layer（P-20260331-001）の制度化により、
ATOSは「観測・判断システム」として確立された。

Phase 9 では `READY → Approval → Execution` の接続を定義し、
ATOSを「判断システム」から「条件付き実行システム」へ移行させる。

本 Proposal は Execution Gate の構造・条件・制御フローを定義する。

---

## 2. Execution Gate の定義

Execution Gate とは、`readiness_state = READY` を受けて
実際の発注（Order送信）を行うか否かを最終決定するレイヤーである。

```
Preview Layer
    ↓ readiness_state = READY
Execution Gate  ← 本 Proposal の定義範囲
    ↓ approved
Order Execution
```

---

## 3. Gate 通過条件（提案）

Execution Gate を通過するために必要な条件：

### 3.1 必須条件（AND）

| 条件 | 説明 |
|---|---|
| `readiness_state = READY` | Preview Layer の判定が READY であること |
| `Founder Approval` | Founder による明示的な実行承認 |
| `session_within_trading_hours` | 取引時間内（9:00〜15:25）であること |
| `no_open_position` | 同一 symbol に未決済ポジションがないこと（他 symbol のポジションは Gate 判定に影響しない） |

### 3.2 禁止条件（いずれかに該当すれば Gate 不通過）

| 条件 | 説明 |
|---|---|
| Hard Limit 違反 | P-20260330-001 定義の Hard Limits に抵触 |
| `execution_performed = true` | 同セッション内で既に実行済み |
| データ鮮度 > 60秒 | Gate 通過直前に再チェック |

---

## 4. Approval フロー（提案）

```
1. Preview → READY 確認
2. Proposer → Founder に実行承認を申請
3. Founder が明示的に "承認" を表明
4. Execution Gate 通過
5. Order 送信
6. trace_event: execution_performed（event_type新設）
```

- Approval は **毎回** Founder が明示的に行う（自動承認なし）
- 承認有効期限: 当該 READY 判定から **60秒以内**（Founder Directive — Preview freshness 上限と整合）
- 60秒を超過した場合は Approval 無効とし、Preview 再実行を要する

---

## 5. trace_event 拡張（提案）

Phase 9 で新設するイベントタイプ：

| event_type | 発火タイミング |
|---|---|
| `execution_gate_approved` | Founder 承認完了時 |
| `execution_gate_rejected` | Gate 不通過時（理由記録） |
| `execution_performed` | Order 送信完了時 |
| `execution_failed` | Order 送信失敗時 |

## 6. 実装スコープ（Phase 9）

### Phase 9A — Gate 定義（制度）
- 本 Proposal の Founder 承認
- Approval フローの文書化
- KabuStation 発注 API は**未接続**（Phase 9A スコープ外）

### Phase 9B — Gate 実装（技術）
- `execution_gate.py` 新規作成
- Gate 通過条件の実装・テスト
- trace_event 新イベントタイプの実装
- `simulated_order` による動作検証
- KabuStation 発注 API エンドポイント・認証方式の検証（Phase 9B で初接続）
- **Phase 9A → 9B 移行条件: `simulated_order` 3回連続成功 かつ trace_event 整合確認**

### Phase 9C — Execution 接続
- KabuStation API 実発注エンドポイントへの本接続
- `execution_performed = true` の記録
- エラーハンドリング・ロールバック設計
- Phase 9C 着手前に独立した Proposal を要する

---

## 7. 制度的制約

- **Execution は常に Founder の明示的承認を要する**（自律実行なし）
- **Gate 通過条件の変更は Founder Directive レベルの承認を要する**
- **Phase 9A 完了前に Phase 9B・9C に着手しない**
