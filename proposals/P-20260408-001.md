## 1. Background

現在の OpenClaw 運用では、以下の三要素が同一スレッドに混在している。

```
Conversation（対話ログ）
Execution（sendorder / shell execution）
Audit（trace_event）
```

この構造は以下の問題を引き起こす。

- 実行仕様が会話に埋没する
- 実行の再現性が保証されない
- 実行と記録の境界が曖昧になる
- STOP の制度的扱いが不明確になる

Founder により提案された「三段分離」：

```
Conversation → Execution → Audit
```

を実現するため、第一段階として **Task Artifact の制度定義** を行う。

---

## 2. Objective

本 Proposal は以下を定義する。

```
A. OpenClaw の実行単位を「Task Artifact」として制度化
B. Conversation と Execution の分離
C. Execution と Audit の分離
D. 実行の再現性・追跡性・停止規律の強化
```

---

## 3. Definition

### 3.1 Task Artifact とは

Task Artifact とは：

```
bounded execution を定義する制度的実行単位
```

であり、以下を含む。

- 実行コマンド
- 期待出力形式
- STOP 条件
- 実行前提条件

---

## 4. Artifact Format（要件 1）

### 4.1 基本形式

Task Artifact は以下形式を採用する。

```
.md（YAML front matter + Markdown 本文）
```

### 4.2 構造

```markdown
---
task_id: step-1-boundary-validation
version: 1.0
role: collector
step: 1
command: |
  bash -lc '...'
expected_output:
  - PRECONDITION CHECK
  - COMMAND
  - RESULT
stop_conditions:
  - boundary_violation
  - path_mismatch
writes_state: false
requires_approval: false
parent_document: OPENCLAW_FLIGHT_CHECKLIST.md
---

# Step 1 — Boundary Validation

## Purpose
...

## Notes
...
```

### 4.3 原則

```
front matter は machine-readable
本文は human-readable
command は単一実行単位（atomic）
```

---

## 5. Parent-Child Relation（要件 2）

### 5.1 親タスク

Conversation（対話ログ）は制度上：

```
Task Planning Layer として扱う
```

### 5.2 子タスク

Task Artifact は以下の子タスクを生成する。

| 子タスク | 役割 |
|---------|------|
| Execution Task | 実行 |
| Audit Task | 記録 |

### 5.3 関係構造

```
Conversation
  ↓
Task Artifact（定義）
  ↓
Execution（bounded）
  ↓
Audit（trace_event / result artifact）
```

### 5.4 原則

```
Conversation は実行しない
Task Artifact のみが Execution を定義する
Execution は Task Artifact に完全従属する
```

---

## 6. md / json Split（要件 3）

### 6.1 分離原則

```
定義 → .md
結果 → .json
```

### 6.2 .md（定義）

用途：
- task specification
- human review
- version control

### 6.3 .json（結果）

用途：
- execution result
- machine validation
- trace linkage

### 6.4 Result Artifact 例

```json
{
  "task_id": "step-1-boundary-validation",
  "status": "completed",
  "sections": {
    "PRECONDITION CHECK": "OK",
    "COMMAND": "...",
    "RESULT": "OK"
  },
  "timestamp": "2026-04-08T..."
}
```

## 7. Task Lifecycle（要件 4）

Task は以下の状態を持つ。

```
pending → active → completed
                 ↓
              aborted
```

### 7.1 状態定義

| 状態 | 意味 |
|------|------|
| `pending` | 未実行 |
| `active` | 実行中 |
| `completed` | 正常終了 |
| `aborted` | STOP または失敗 |

### 7.2 遷移原則

```
pending → active:    明示的トリガー
active → completed:  成功
active → aborted:    STOP または failure
```

### 7.3 不可逆性

```
completed / aborted は終端状態
終端状態からの復帰は不可
```

---

## 8. STOP Recording Policy（要件 5）

### 8.1 STOP の定義

STOP は：

```
規律ある実行停止
であり失敗ではない
```

### 8.2 STOP 時の処理

STOP 発生時：

1. Task は `aborted` に遷移
2. Result Artifact を生成
3. trace_event を記録

### 8.3 Artifact 封印

STOP 発生時：

```
Task Artifact は変更禁止
Result Artifact は確定状態として保存
```

### 8.4 trace_event

```json
{
  "event_type": "step_stopped",
  "task_id": "...",
  "reason": "...",
  "step": 5
}
```

## 9. Execution Model との整合

本 Proposal は以下と整合する。

| 制度 | 整合点 |
|------|-------|
| Three-Layer Model | Execution は WRITE のみ |
| Execution Attribution Principle | 実行は bounded agent に帰属 |
| Authorization Lifecycle | active → consumed / invalidated / expired |
| Execution Safety（READY 消費モデル） | 1 READY = 1 execution |

---

## 10. Non-Goals

本 Proposal は以下を扱わない。

```
・ sendorder 実行仕様
・ AAB 詳細定義
・ Execution Agent 実装（runner）
```

## 11. Expected Impact

本 Proposal により：

1. **実行仕様の Git 管理** — Task Artifact が制度文書として管理される
2. **実行の再現性確立** — command が artifact に封じられる
3. **STOP の制度成果化** — 停止もまた artifact として記録される
4. **OpenClaw の役割再定義** — 「会話主体」から「Execution 管理主体」へ移行
