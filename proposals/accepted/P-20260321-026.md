## Block-driven Hypothesis Layer

**Date:** 2026-03-21
**From:** Librarian (GPT)
**Reviewed by:** Proposer (Claude)
**Status:** Rev.1 — Ready for Acceptance Discussion
**Type:** Interpretation Layer
**Constitutional Impact:** None

---

## 1. Background

P-024 established execution_blocked as a first-class event.
P-025 introduced taxonomy and structured interpretation of block reasons.

However, blocked events are not yet used to generate institutional learning.

---

## 2. Objective

Introduce a system where execution_blocked events become inputs for hypothesis generation.

This proposal defines:

- block aggregation model
- hypothesis trigger conditions
- proposer interface for block-driven reasoning

---

## 3. Core Concept

execution_blocked is not a failure.
It is a signal.

System flow:

```
execution_blocked
→ classification (P-025)
→ aggregation
→ proposer hypothesis
→ proposal
→ execution model improvement
```

---

## 4. Aggregation Model

Minimum aggregation dimensions:

- block.code
- block.category
- block.severity
- attempted_task
- linked_gate_event_id
- linked_preview_event_id
- ts (timestamp)
- time window

---

## 5. Hypothesis Trigger Conditions

Hypothesis should be triggered when:

- same block.code repeats
- same category dominates a time window
- specific lineage repeatedly fails

Threshold definition is out of scope for P-026.
Proposer applies judgment-based interpretation
until quantitative thresholds are established in a future Proposal.

Example:

- B-4 repeated → authorization constraint too strict?
- B-6 repeated → slippage model inaccurate?
- B-5 clustered → liquidity evaluation insufficient?

---

## 6. Proposer Interface

Proposer receives:

- aggregated block summary
- distribution by code/category
- temporal patterns

Proposer responsibilities:

① generate hypothesis from repeated block patterns
② assess if block is desirable or excessive
③ produce Proposal (through standard proposal process)
   for execution model or policy improvement
   — hypothesis alone does not constitute a Proposal

---

## 7. Hypothesis Output (Example)

```json
{
  "proposal_type": "execution_improvement",
  "trigger": "block_pattern",
  "block_code": "B-4",
  "hypothesis": "authorization constraints may be overly restrictive",
  "confidence": "medium"
}
```

## 8. Institutional Role Separation

| Role | Responsibility |
|------|---------------|
| Collector | records execution_blocked |
| Librarian | ensures taxonomy and aggregation integrity |
| Proposer | interprets patterns and generates hypotheses |
| Registrar | executes approved improvements |

---

## 9. Expected Effect

P-026 transforms:

```
execution_blocked → knowledge source
```

This enables:

- adaptive execution policy
- feedback-driven system improvement
- measurable learning loop

---

## 10. Conclusion

P-024 proved the system can stop.
P-025 explained why it stopped.
P-026 allows the system to learn from stopping.
