## 1. Background

ATOSは以下の制度フローを持つ：

```
Observation → Dialogue → Proposal → Preview → Approval → Execution
```

これまでのテストにより：
- Observation → Preview → Gate の再現成功
- stale observation に対する NOT_READY 判定の正常動作
- Execution の適切な抑止
- trace_event の一貫した記録

が確認された。

しかし、これらは個別テストとして成立しているに過ぎず、
開場中に安定して運用するための標準手順（Operational Protocol）が未定義である。

## 4. ATOS Operational Protocol v1

### 4.1 実行シーケンス（最小構成）

```
1. Preflight
2. Go / No-Go 判定
3. Command Classification Declaration
4. Market Observation
5. Observation Freshness Check
6. Preview / Readiness Check
7. Execution Gate Confirmation
8. Bounded WRITE Execution（必要時のみ）
9. Report
```

### 4.2 各ステップ定義

**Step 1. Preflight**

- runtime base / venv / env / PowerShell bridge の成立確認
- 確認項目：
  - OpenClaw workspace が正しい base にあること（`/mnt/c/ai-trading-os-private`）
  - `.env.local` が同一シェルコンテキストで source 済みであること
  - `.venv` が同一シェルコンテキストで activate 済みであること
  - Windows PowerShell が PATH に含まれていること（`powershell.exe` 解決可能）
- 復旧が必要な場合は `docs/anchors/technical/OBSERVATION_PREVIEW_RUNTIME_RECOVERY_ANCHOR.md` を参照すること

**Step 2. Go / No-Go 判定**

- **いずれかの**前提条件が成立しない場合は即停止
- 停止後は Preflight に戻り復旧を試みる
- 復旧不能の場合は Founder に報告して終了

**Step 3. Command Classification Declaration**

- 実行予定コマンドを以下に分類して先出し：
  - `observation-only`：READ のみ、外部状態変更なし
  - `bounded WRITE`：trace_event 書き込みを伴う、スコープ明示必須

**Step 4. Market Observation**

- 最新市場データの取得（`collect_board_once.py` または同等）
- `market_observation` イベントを trace_event に記録

**Step 5. Observation Freshness Check**

- データが最新であることを確認（freshness_seconds < 60）
- stale（≥ 60秒）の場合は即停止 → `DATA_INVALID` として記録

**Step 6. Preview / Readiness Check**

- Hard Limits / Soft Limits / データ整合性を評価
- 状態を以下の4値で判定（P-20260331-001 確定値）：
  - `READY`
  - `NOT_READY`
  - `REPRICE_REQUIRED`
  - `DATA_INVALID`
- `execution_readiness_evaluated` として trace_event に記録

> **注:** `HUMAN_REVIEW_REQUIRED` は P-20260328-017 に定義されていたが、
> P-20260331-001 の正規化により現行実装では上記4値に統合されている。
> 将来的な再定義が必要な場合は別途 Proposal を要する。

**Step 7. Execution Gate Confirmation**

- 実行するかどうかの境界を明示：
  - `NO_GO`（停止）：READY 以外、または Founder 未承認
  - `bounded WRITE 実行許可`：READY かつ Founder 明示承認（TTL 60秒以内）
- Founder 承認を Approval Layer として位置づける（六段フローにおける Approval に相当）

**Step 8. Bounded WRITE Execution**

- 許可された単一スコープのみ実行
- すべて trace_event を伴う
- Kill Switch（`KILL_SWITCH.flag`）の存在確認を最優先で実施

**Step 9. Report**

- 短報（毎回必須）：
  - observation 成否
  - freshness（秒）
  - readiness_state
  - gate 結果
  - WRITE 有無

## 5. Core Principles

### 5.1 Freshness Principle

```
Preview は常に最新 observation を前提とする。
stale observation（≥ 60秒）では readiness 判定を行わない。
```

### 5.2 Execution Boundary Principle

```
Execution は必ず gate を通過した後にのみ発生する。
```

### 5.3 Command Classification Principle

```
すべてのコマンドは実行前に
- observation-only
- bounded WRITE
のいずれかに分類されなければならない。
```

### 5.4 Minimal Execution Principle

```
WRITE は必要最小限に限定される。
```

### 5.5 Kill Switch Principle（P-20260331-003 継承）

```
KILL_SWITCH.flag が存在する場合、
Execution Gate は他の全条件に優先して NO_GO を返す。
```
