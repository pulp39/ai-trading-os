**Scope:** preview → execution_gate_passed まで。real_order は含まない。

---

## 1. 背景

P-20260321-020（trace_event: 272）により execution_gate が制度的に定義された。
P-20260321-021（trace_event: 273）により execution_gate が操作可能になった。

しかし現時点で：

```
execution_gate_passed は一度も発行されていない
```

つまり：

```
Gateは「押せるボタン」になったが、まだ誰も押していない
```

本 Proposal は、execution_gate を初めて通過させる**制御されたテスト**である。

---

## 2. 本 Proposal の性質

> **本 Proposal は「設計」ではなく「検証」である。**

設計はP-020・P-021で完了している。本 Proposal が問うのは：

- Gateは実際に通過できるか
- trace_event構造は正しく生成されるか
- 実行前最終境界は安定しているか

の3点のみである。

より正確に言えば：

```
「Gateを通す」ことではなく
「Gateが通る世界を証明する」こと
```

---

## 3. 目的

| # | 検証目的 |
|---|---------|
| V-1 | `execution_gate_passed` trace_event が正しく生成・記録されること |
| V-2 | precondition 2層構造（宣言層・実装層）が正しく機能すること |
| V-3 | Gate通過後も execution が自動発生しないこと（I-9の確認） |
| V-4 | `linked_preview_event_id` による preview との連鎖が正しく記録されること |

## 6. 実行手順

### Root Condition（前提確認）

本テストにおける `execution_gate_passed` の生成は、
execution の根拠そのものではなく：

```
「execution 可能状態の成立」を示す唯一の制度イベント
```

である。

すなわち：

```
Gate = execution 可能性の成立
Gate ≠ execution の発動
```

この区別は本テスト全体を通じて保持される。

---

### Step 1 — execution_gate_task.json の作成

```json
{
  "task_type": "execution_gate",
  "date": "2026-03-21",
  "authorized_by": "claude_registrar",
  "linked_preview_event_id": 269,
  "scope": "bounded",
  "preconditions": {
    "requires_preview": true,
    "requires_determinism": true,
    "requires_role_boundary_check": true,
    "requires_scope_clarity": true
  },
  "metadata": {
    "gate_purpose": "First controlled verification test of execution_gate passage (P-022)",
    "real_order_authorized": false
  }
}
```

---

### Step 2 — precondition チェック（実装層）

スクリプトが以下を順番に確認する：

```
[Root Condition チェック]
① 同一 preview_event_id（269）に対して既存の execution_gate_passed が
   存在しないこと（I-6：One Gate per Preview）
   └─ これが失敗した場合、他の条件に進まず即停止

[Integrity Conditions チェック]
② research.trace_event に id=269（order_preview_recorded）が存在するか
③ 対象 preview が deterministic か（同一入力で再現可能か）
④ role boundary violation が存在しないか
⑤ execution scope が明確か（task に明示されているか）
```

いずれかが失敗した場合 → `execution_blocked` を記録して停止。

---

### Step 3 — execution_gate_passed の挿入

全preconditionが通過した場合のみ、以下を `research.trace_event` に挿入：

```json
{
  "event_type": "execution_gate_passed",
  "agent_id": "claude_registrar",
  "summary": "Execution gate passed — first controlled verification test (P-022)",
  "metadata": {
    "linked_preview_event_id": 269,
    "gate_task_ref": "registrar_queue/execution_gate_task.json",
    "authorized_by": "claude_registrar",
    "gate_role": "registrar_authorization",
    "preconditions_verified": true,
    "real_order_authorized": false,
    "test_context": "P-20260321-022 controlled verification"
  }
}
```

---

### Step 4 — 結果確認

挿入後、以下を確認する：

```
① trace_event_id が採番されたか
② linked_preview_event_id = 269 が正しく記録されたか
③ real_order_authorized = false が保持されているか
④ execution が発生していないこと（I-9の確認）
⑤ 同一 preview_event_id に対する gate が1件のみであること（I-6の確認）
```

---

## 8. 失敗条件と対応

| 失敗条件 | 対応 |
|----------|------|
| Root Condition チェック失敗（既存gate発見） | 即停止。既存 `execution_gate_passed` の確認が必要 |
| Integrity Condition チェック失敗 | `execution_blocked` を記録。原因を特定してP-022の再実行を検討 |
| trace_event 挿入失敗 | DBアクセス問題として技術調査。制度的問題ではない |
| real_order が誤って発生 | 重大な制度違反。即時停止・Founder報告 |

---

## 9. Invariant 確認（本テストで検証対象）

| # | Invariant | 本テストでの確認方法 |
|---|-----------|-------------------|
| I-5 | Gate 単一性 | `execution_gate_passed` のみが生成されることを確認 |
| I-6 | One Gate per Preview | id=269 に対してgateが1件のみであることを確認（Root Condition） |
| I-7 | Gate is reproducible | 同じ条件で再実行可能な構造であることを確認 |
| I-8 | Gate is irreversible | 挿入後にtrace_eventが変更されないことを確認 |
| I-9 | Gate does not trigger execution | Gate通過後にreal_orderが発生しないことを確認 |
