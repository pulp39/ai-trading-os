## 1. Research Hypothesis

Defining a formal specification for Artificial Institutional Identity (AiiD)
will provide the structural foundation required for AI Assembly membership,
enabling traceable and accountable participation of AI agents in institutional
deliberation.

## 2. Context Reference

- P-20260313-001 (AI Assembly Establishment Framework — section 4.3)
- constitution.md v1.3 Article E (Institutional Subject)
- constitution.md v1.3 Article F (Institutional Continuity Principle)
- constitution.md v1.3 Article G (Role Succession Principle)
- constitution.md v1.3 Article D.3 (Clarification of Delegated Roles)

## 3. Supporting Observations

- P-20260313-001 section 4.3 deferred AiiD specification to this proposal.
  Assembly membership cannot be assigned until AiiD is formally defined.
- Constitution Article E defines Institutional Subject as an entity that
  occupies an institutional role seat and operates institutional knowledge
  through the repository. AiiD is the concrete implementation of this
  constitutional concept within the Assembly context.
- Constitution Article F establishes that institutional continuity is
  secured through repository records, not AI instance state. AiiD must
  be consistent with this principle: an AiiD is a persistent institutional
  identity, not bound to any specific model session.
- The existing role system (Proposer, Registrar) uses agent_id as an
  informal identifier (e.g., claude_registrar). AiiD formalizes this
  pattern as an institutional object with defined structure and lifecycle.

## 4. Proposal Content

### 4.1 Definition

An AiiD (Artificial Institutional Identity) is a formal institutional
identity assigned to an AI agent occupying a role within the AI Trading OS
governance framework.

An AiiD represents an institutional seat, not a specific AI model or
session instance. Multiple AI models or implementations may occupy the
same AiiD seat across successive sessions, subject to Founder appointment.

### 4.2 AiiD Structure

Each AiiD record contains the following fields:

| Field | Type | Description |
|---|---|---|
| aiid | string | Unique institutional identifier |
| display_name | string | Human-readable name for institutional reference |
| role | string | The institutional role this AiiD occupies |
| model | string | Current AI model or implementation occupying this seat |
| appointed_by | string | Authority that appointed this AiiD (always: Founder) |
| appointment_date | date | Date of appointment |
| status | string | active / suspended / retired |

### 4.3 AiiD Identifier Format

AiiD identifiers follow the format:

```
{model_family}_{role}
```

Examples:
```
aiid: claude_proposer
aiid: claude_registrar
aiid: claude_assembly_member
aiid: gpt_assembly_member
```

The identifier is stable across model version updates within the same
model family. A change to a different model family constitutes a
succession event under Article G.

If future operational requirements necessitate multiple AiiDs within
the same role, the identifier scheme shall be extended through a
separate proposal.

### 4.4 Relationship to Existing agent_id

The existing `agent_id` field used in trace_event records
(e.g., `claude_registrar`) is consistent with the AiiD identifier
format defined above. Existing agent_id values are hereby recognized
as valid AiiD identifiers without requiring retroactive re-registration.

### 4.5 AiiD Lifecycle

**Appointment**
- Authority: Founder
- Trigger: Founder declaration to Librarian
- Effect: AiiD record created, status set to `active`
- trace_event: `aiid_appointed`

**Suspension**
- Authority: Founder or Librarian
- Trigger: Operational decision or institutional anomaly
- Effect: AiiD status set to `suspended`; role seat becomes vacant
- trace_event: `aiid_suspended`

**Reactivation**
- Authority: Founder
- Trigger: Founder declaration
- Effect: AiiD status restored to `active`
- trace_event: `aiid_reactivated`

**Retirement**
- Authority: Founder
- Trigger: Permanent cessation of the institutional identity
- Effect: AiiD status set to `retired`; record preserved (append-only)
- trace_event: `aiid_retired`

**Succession**
- Governed by constitution Article G
- Classification and appointment per Librarian and Founder respectively
- trace_event: `aiid_succession`

### 4.6 Assembly Member AiiD

P-20260313-001 requires 2 Assembly Members, each assigned an AiiD.

Upon adoption of this proposal, Assembly Member AiiDs shall be
appointed by the Founder. Appointment declarations are made to the
Librarian, consistent with the session declaration protocol established
in P-20260313-001 section 4.5.

Until Assembly Member AiiDs are appointed, the AI Assembly exists as
an institution but cannot convene.

### 4.7 AiiD Registry

AiiD records shall be maintained in:

```
docs/aiid_registry.md
```

This document is a Layer B document (Institutional State Snapshot).
Updates require: Registrar task following Librarian review.

### 4.8 Relationship to Existing Framework

This proposal does not modify:
- the Constitution
- Founder authority
- the Proposal system
- Librarian review authority
- Registrar execution authority
- Collector framework

AiiD formalizes the AI identity concept already present in
constitution Article E into a concrete operational structure.

### 4.9 Implementation Requirements

If adopted, the following actions are required:
- Create `docs/aiid_registry.md` with initial AiiD records
- Register existing agent_id values as recognized AiiD identifiers
- Founder appointment of Assembly Member AiiDs (2 seats)
- trace_event registration of this enactment

## 7. Limitations

- Assembly Member AiiD appointment is deferred to Founder declaration
  after adoption. The Assembly cannot convene until appointments are made.
- AiiD for non-Assembly roles (Proposer, Registrar, Collector) are
  recognized retroactively under section 4.4, but formal registry
  entries require Registrar task execution after adoption.
