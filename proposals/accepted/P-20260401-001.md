## 1. Background

### 1.1 Phase 9B における欠陥の発見

Phase 9B（simulated_order接続）の動作検証において、以下の Critical 欠陥が確認された。

```
同一 READY context において、複数回の execution_performed が記録可能である。
```

現行実装では：
- `execution_gate.py` は `readiness_state = READY` を受けて Gate 通過を判定する
- しかし、同一の Preview 判定（同一 READY context）に対し、Gate が複数回通過できる
- その結果、Founder 承認が1回であっても、複数回の発注が技術的に可能な状態にある

### 1.2 制度的問題

P-20260331-002（Execution Gate）Section 3.2 は以下を禁止条件として定義している：

```
execution_performed = true：同セッション内で既に実行済み
```

しかしこれは「セッション単位」での制御であり、「READY context 単位」での消費管理が存在しない。

すなわち：
- 複数の READY context が同一セッション内で生成されれば、複数回の execution が可能
- 同一 READY context の再利用（Replay）を防ぐ仕組みが未定義

### 1.3 real_order 接続前の制度化が必要

Phase 9B-3（実発注テスト・P-20260331-003）着手前に、本欠陥を制度的に閉じなければならない。

```
本 Proposal は Phase 9B-3 着手の前提条件として位置づけられる。
```

---

## 2. Objective

```
1 READY context = 1 execution という原則を制度として定義し、
Duplicate Execution を制度的・技術的に防止する。
```

---

## 3. Core Principle

```
READY context は消費可能（Consumable）なリソースである。
一度 consumed となった READY context は、いかなる条件下でも再利用できない。
```

## 4. Proposal

以下のモデルを ATOS Execution Safety Layer に追加定義する。

## 5. Execution Consumption Model

### 5.1 READY Context Identity

READY 判定が発生した時点で、当該判定に一意な識別子を付与する。

| 項目 | 定義 |
|------|------|
| `ready_context_id` | READY 判定ごとに生成される UUID（例: `rctx-20260401-0001`） |
| 生成タイミング | `execution_readiness_evaluated` イベント記録時（`readiness_state = READY` の場合のみ） |
| スコープ | 当該 READY 判定から生成された execution が完了または失敗するまで有効 |
| 形式 | `rctx-YYYYMMDD-NNNN`（日付 + 4桁連番） |

`ready_context_id` は以下のイベントに付与される：
- `execution_readiness_evaluated`（生成元）
- `execution_gate_approved`
- `execution_gate_rejected`
- `execution_performed`
- `execution_failed`
- `execution_consumed`
- `execution_blocked`

### 5.2 Consumption State

READY context は以下の状態を持つ。

| 状態 | 意味 | 遷移条件 |
|------|------|----------|
| `active` | 消費前・実行可能 | READY 判定直後に設定 |
| `consumed` | 正常消費・再利用不可 | `execution_performed` 記録時 |
| `invalidated` | 無効化済み | `execution_failed` / Freshness 超過 / Kill Switch 発動 / 異常時 |
| `expired` | TTL 超過 | READY 判定から 60秒経過（P-20260331-002 TTL と整合） |

**状態遷移規則:**

```
active
  → consumed:    execution_performed 発生時（正常消費）
  → invalidated: execution_failed / Kill Switch 発動 / データ異常 / 手動無効化
  → expired:     TTL 60秒超過

※ READY context の expired（文脈有効期限）と Founder Approval TTL（承認有効期限）は概念上独立である。
現段階では両者を 60 秒で整合させるが、将来的に分離される可能性がある。
```

`consumed` / `invalidated` / `expired` からの復帰は不可（一方向遷移）。

### 5.3 Duplicate Blocking

Execution Gate において、以下のチェックを **Kill Switch の次に優先** して実施する。

```
if ready_context.state != "active":
    trace_event: execution_blocked（reason: duplicate:<state>）
    return NO_GO
```

| ブロック条件 | reason 値 |
|-------------|-----------|
| `consumed` 状態の再利用 | `duplicate:consumed` |
| `invalidated` 状態の再利用 | `duplicate:invalidated` |
| `expired` 状態の再利用 | `duplicate:expired` |

### 5.4 Trace Event 設計

本 Proposal により以下の trace_event を新設する。

| event_type | 発火タイミング | 説明 |
|------------|---------------|------|
| `execution_consumed` | Execution 完了後（consumed 遷移時） | 正常消費の記録 |
| `execution_blocked` | Duplicate 検出時 | 重複実行の阻止記録 |

**`execution_consumed` 記録内容:**
```json
{
  "event_type": "execution_consumed",
  "ready_context_id": "<対応する ready_context_id>",
  "consumed_by": "<execution_performed の event_id>",
  "timestamp": "<ISO8601>"
}
```

**`execution_blocked` 記録内容:**
```json
{
  "event_type": "execution_blocked",
  "ready_context_id": "<対応する ready_context_id>",
  "reason": "duplicate:<detail>",
  "blocked_at": "<ISO8601>"
}
```

### 5.5 Context Invalidation（異常時 Freeze）

以下の条件が発生した場合、active な READY context は即時 `invalidated` へ遷移する。

| 条件 | 処理 |
|------|------|
| Kill Switch 発動（KILL_SWITCH.flag 存在確認） | 全 active context → invalidated |
| Freshness 再チェック失敗（Gate 直前チェック）| 当該 context → invalidated |
| Hard Limit 再評価で違反検出 | 当該 context → invalidated |
| Proposer/システムによる手動無効化 | 当該 context → invalidated |

## 6. Execution Gate への統合

本モデルの統合後、Execution Gate の評価順序は以下となる。

```
1. Kill Switch チェック（最優先・P-20260331-003 継承）
2. READY Context State チェック（本 Proposal 追加）
   → state != active → execution_blocked → NO_GO
3. Founder Approval 確認（TTL 60秒・P-20260331-002）
4. 禁止条件チェック（Freshness / Hard Limit / execution_performed）
5. Gate 通過 → execution_gate_approved → Order Execution
6. execution_performed 記録 → execution_consumed 記録（context: consumed）

※ execution_performed と execution_consumed は論理的に不可分であり、実装上は同一トランザクションとして扱うこと。
```

## 8. Core Principles

### 8.1 Consumption Principle

```
1 READY context = 1 execution
READY context は一度しか消費できない。
```

### 8.2 Non-Reversibility Principle

```
consumed / invalidated / expired 状態からの復帰は不可。
新たな execution が必要な場合は、新たな Observation → Preview サイクルを要する。
```

### 8.3 Audit Completeness Principle

```
すべての Duplicate 阻止は execution_blocked として記録される。
阻止された事実が監査証跡に残らない Duplicate 防止は制度として不完全である。
```

### 8.4 Gate Integration Principle

```
Consumption チェックは Execution Gate の一部として実行される。
Preview Layer の外部に独立した Consumption レイヤーを設けない。
```

## 10. Impact

本 Proposal により：
1. **Duplicate Execution の制度的閉鎖** — 同一 READY context の再利用を完全に防止
2. **real_order 接続の安全性確保** — Phase 9B-3（実発注）の前提条件を満たす
3. **監査可能性の向上** — `execution_consumed` / `execution_blocked` による完全な追跡
4. **Replay Attack 防止** — 承認済み Gate 状態の再利用による不正実行を防止
5. **TTL 整合** — Approval 有効期限（60秒）との一貫した時間管理
