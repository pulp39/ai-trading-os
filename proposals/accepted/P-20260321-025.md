---

## 1. Background

P-20260321-024 established the first real execution_blocked event.
This verified that stopping is a first-class institutional act.

However, the current blocked-path representation still has two limits:

- rejection_reason is represented as a single flat string
- verification can overcount false positives when matching textual content

The P-024 verification incident demonstrated the second issue clearly:
a legacy verification query using `content ILIKE '%real_order%'` returned NG
because prior execution_recorded rows merely mentioned real_order in explanatory
text while explicitly recording `real_order_triggered=false`.

---

## 2. Objective

This proposal introduces:

- a structured block reason taxonomy (with category separation)
- scoped verification based on metadata lineage rather than free-text search
- a proposer-facing interpretation bridge so blocked execution becomes analyzable
  institutional knowledge

---

## 3. Why This Matters

P-022 proved that the system can pass a gate.
P-024 proved that the system can stop by design.
P-025 makes blocked execution legible, classifiable, and reviewable.

Without this layer, `execution_blocked` remains visible but semantically weak.
With this layer, blocked events become usable for analytics, review, and proposer-led improvement.

---

## 4. Scope

**Included:**

- taxonomy definition for block reasons (with category separation)
- metadata shape for execution_blocked (with reserved severity field)
- refined verification rules for blocked-path tests (lineage-scoped)
- proposer-facing aggregation model for future interpretation

**Not included:**

- live real_order implementation

| B-1 | scope_violation | governance |

## 6. Metadata Model (Backward Compatible + Structured)

Recommended metadata shape:

```json
{
  "linked_gate_event_id": 275,
  "linked_preview_event_id": 269,
  "rejection_reason": "B-4_real_order_not_authorized",
  "attempted_task": "real_order",
  "execution_intent": true,
  "blocked_by_design": true,
  "block": {
    "code": "B-4",
    "category": "authorization",
    "reason": "real_order_not_authorized",
    "severity": "hard"
  }
}
```

**Notes:**
- backward compatibility is preserved
- severity is reserved for future use:
  - `hard`: absolute block (no retry possible)
  - `soft`: condition-dependent (retry possible)

---

## 7. Scoped Verification Principle

Verification must not rely on free-text matching.

**Old anti-pattern:**
```
content ILIKE '%real_order%'
```

**Refined principle:**
- event_type must match execution class
- attempted_task must be read from metadata
- lineage must be restricted to a defined chain

**Lineage chain definition:**
```
(parent_event_id, linked_gate_event_id, linked_preview_event_id)
```

---

## 8. Verification Rule

No real execution exists when:

- no `execution_recorded` row exists for `attempted_task = real_order`
- within the lineage chain defined above

## 9. Proposer Interface (Minimum Definition)

**Aggregation dimensions:**

- block.code
- block.category
- block.severity (future)
- attempted_task
- linked_gate_event_id
- linked_preview_event_id
- symbol (when applicable)
- time bucket

**Minimum proposer use-cases:**

① Block reason aggregation reporting
② Hypothesis drafting when block.code repeats
③ Registrar-facing improvement proposals

This establishes the bridge to block-driven hypothesis generation.
