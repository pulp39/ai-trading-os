## 1. 背景

現在の制度状態：

```
market_observation   → operational
simulated_order      → operational
order_preview        → operational（off-hours含む）
real_order           → not yet implemented
```

preview層はすでに deterministic・reproducible・testable であり、
「実行直前の完全な観測状態」が成立している。

しかし：

> **preview → real_order の遷移に制度的ガードが存在しない**

これは以下の憲法上リスクを内包する：

- proposal の非実行性侵害
- role collapse（Proposer → Execution）
- traceability 断絶

---

## 2. 目的

real_order 実行に対して、制度的に明確で不可侵な境界を導入する。

---

## 3. Conceptual Layer

### 3.1 execution_gate 定義

**execution_gate** とは：

> preview 結果を execution へ昇格させるための唯一の制度的通過点

性質：

- **非自動**（必ず明示的通過）
- **単一入口**（single choke point）
- **audit 可能**
- **role-bound**（Registrar のみが開く）

### 3.2 Flow 上の位置

```
market_observation
        ↓
proposal（解釈）
        ↓
simulated_order
        ↓
order_preview（非実行）
        ↓
⛔ execution_gate        ← 本 Proposal が定義する境界
        ↓
real_order（未実装）
```

**重要：execution_gate は preview と execution の間にのみ存在する**

### 3.3 Gate の制御主体

execution_gate は **Registrar 経由でのみ開く**。

| ロール | 権限 |
|--------|------|
| Proposer | advisory のみ |
| Librarian | 解釈・整合性のみ |
| Collector | 観測のみ |
| **Registrar** | **execution_gate を開く唯一の主体** |

## 4. Execution Layer

### 4.1 Execution Authorization Model（確定構造）

実行の論理構造：

```
Execution = Gate  WITH  Integrity Constraints
```

すなわち：

- **Gate が存在しない → Execution 不可能**（他の条件がすべて満たされても）
- **Gate が存在する → Execution 可能**（ただし Integrity Conditions で停止し得る）

#### 必須条件（構造定義）

```
[Root Condition]
① execution_gate_passed trace_event の存在
   └─ これが実行の唯一の制度的根拠

[Integrity Conditions]
② linked_preview_event_id の存在と整合性
③ Registrar task としての明示的実行指示
④ role boundary violation が存在しないこと
⑤ execution scope が明確であること
```

> ※ Root Condition が存在しない場合、Integrity Conditions がすべて満たされても実行は不可能

### 4.2 trust_token / AAB の扱い

- **AAB（Activity Authorization Bit）は現状維持**
- execution 専用フィールドの追加は**行わない**（現段階）

理由：

- AAB は「活動可能かどうか」を示す層
- execution_gate は「実行してよいか」を判断する制度行為
- これらは本質的に異なる層であり、混在させると Gate の純度が低下する
- execution_gate_passed のみを実行の根拠とする原則を保護するため

### 4.3 Preview との分離

不変条件：

```
proposal ≠ execution
preview  ≠ execution
```

- proposal は「解釈」であり実世界への影響を持たない
- preview は「仮想状態」であり実世界への影響を持たない
- いずれも execution_gate を経ずに real_order を引き起こしてはならない

## 5. Governance Layer

### 5.1 なぜ proposal は実行できないか

憲法より：

```
proposal = advisory
execution ≠ research
```

したがって `proposal → execution` は制度違反。

### 5.2 preview の位置づけ

preview は execution に最も近いが、依然として非実行領域である。

役割：
- 状態確定
- 価格確定
- 再現性確保

### 5.3 execution_gate の意味

```
execution_gate = 「研究」から「行動」への唯一の断層
```

これは単なる技術的制約ではなく：

> **AI Trading OS における「意思決定の倫理的境界」**

## 6. Execution Artifact Definition

### 6.1 新規 event_type：`execution_gate_passed`

Registrar が gate を通過させたことを記録する。

```json
{
  "event_type": "execution_gate_passed",
  "actor_type": "registrar",
  "summary": "Execution gate passed for real_order",
  "details": {
    "preview_event_id": <linked preview trace_event id>,
    "validation": "confirmed",
    "scope": "bounded",
    "execution_ready": true
  }
}
```

### 6.2 既存 event_type 拡張：`execution_recorded`

`execution_recorded` を拡張し、real_order サブタイプとして利用する（新規 event_type は作成しない）。

```json
{
  "event_type": "execution_recorded",
  "metadata": {
    "execution_type": "real_order",
    "linked_gate_event_id": <execution_gate_passed の trace_event id>,
    "linked_preview_event_id": <preview の trace_event id>
  }
}
```

> `subtype` ではなく `execution_type` を使用する（意味の明確化）

## 7. Failure / Rejection Model

### 7.1 Block 条件

execution は以下の条件で**必ず停止**する：

| 条件 | 説明 |
|------|------|
| `execution_gate_passed` trace_event が存在しない | **Root Condition 未充足 — 実行不可能** |
| preview が存在しない・不一致 | Integrity Condition 違反 |
| execution scope が不明 | Integrity Condition 違反 |
| role violation の疑い | Integrity Condition 違反 |
| Registrar 指示が存在しない | Integrity Condition 違反 |

### 7.2 rejection 記録

実行がブロックされた場合、以下を `research.trace_event` に記録する：

```json
{
  "event_type": "execution_blocked",
  "summary": "Execution blocked: <reason>",
  "details": {
    "reason": "<具体的な未充足条件>",
    "action": "halted"
  }
}
```
