Type: Governance Framework — Stability Enforcement
Constitutional Impact: None
Scope: 同一draftからの複数TP生成禁止（Phase 3）

Background
P-20260319-008の検証により以下が確認されました：
同一draft（206）から：
  TP-20260319-002（旧）
  TP-20260319-003（P-008適用後）
→ 重複昇格が現在は防止されていない
制度進化の流れ：
重複発生（観測）
↓ P-008で可視化
↓ P-009で封じる（本Proposal）

Core Rule
1つのproposal_draftedから生成できるTPは1つのみ

Change 1 — 実行順序の明確化（必須）
formalize_trading_proposal.pyの実行順序を以下に固定します：
① review check（P-007）← 必ず先
② duplicate check（P-009）← 必ず後
理由： 逆順になると未reviewのdraftでも「TPがないからOK」として通過してしまい、制度崩壊を招きます。

Change 2 — 重複チェックの導入
formalize_trading_proposal.py実行前（reviewチェック後）に以下を必須チェック：
sqlSELECT id
FROM research.trace_event
WHERE event_type = 'trading_proposal'
  AND metadata->>'source_proposal_draft_event_id' = '206'
ORDER BY id DESC
LIMIT 1;
既存TPが存在する場合（修正③反映）：
pythonraise DuplicateElevationError(
  f"Cannot formalize: TP already exists for "
  f"draft_event_id=206 (existing_tp_id=TP-20260319-003)"
)
```

→ 2つ目のTPは生成されません。運用時に既存TP IDが明示されます。
```
