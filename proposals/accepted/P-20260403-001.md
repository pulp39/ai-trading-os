## 1. Background

### 1.1 観測された問題

2026-04-03 のライブマーケットテストにより、以下の二点が確認された。

**A. kabuStation 再起動後のシンボル登録消失**

再起動後、`symbol list = empty` となり、
コレクターが再起動前と同一のシンボルに対して動作しない状態となった。
リカバリーシーケンス（境界検証 → token取得 → symbol再登録 → collector）を
手動で実行することで復旧したが、この手順は制度化されていなかった。

**B. Preview Freshness による READY ブロック**

現行パラメータ `freshness_limit_ms = 5000` のもとで、
ライブデータを使用した正常な Preview 試行が `snapshot_stale` により BLOCK された。

```
failed_checks = ["snapshot_stale"]
freshness_limit_ms = 5000
measured_freshness_ms ≈ 25427
```

システムは制度通りに動作しており、これは実装障害ではない。
問題は制度パラメータと運用手順の未定義である。

### 1.2 制度的位置づけ

本 Proposal は安全性の弱化を目的とするものではない。

```
Execution safety は検証済みである。
残課題は実装ではなく、
Recovery Procedure の明文化と Readiness Parameter の較正である。
```

## 4. Part A — Restart Recovery Procedure Formalization

### 4.1 問題の定義

kabuStation の Symbol List は **再起動をまたいで永続化されない**。

これは kabuStation の仕様上の特性であり、バグではない。
しかし現在、この特性に対応する制度的手順が存在しない。

```
再起動後に symbol 登録なしで collector を実行した場合:
→ board データ取得失敗
→ Observation Layer が機能しない
→ Preview / Gate / Execution のすべてが前提条件不成立となる
```

### 4.2 Symbol List 状態の分類

| 状態 | 定義 | コレクター実行可否 |
|------|------|-----------------|
| `registered` | kabuStation に symbol が登録済み | 可 |
| `empty` | 再起動後・未登録状態 | 不可 |
| `unknown` | 確認未実施 | 不可（確認が必要） |

### 4.3 Restart Recovery Sequence（確定版）

kabuStation 再起動後は、以下のシーケンスを**この順序で**実行しなければならない。

```
Step 1: 境界検証（Boundary Validation）
  → WSL ↔ PowerShell ↔ kabuStation の接続確認
  → port 18080 の到達性確認（portproxy 競合がないこと）

Step 2: Token 取得確認
  → KABU_API_PASSWORD で token 取得
  → ResultCode = 0 を確認
  → 失敗時は portproxy チェック（RUNTIME_ENVIRONMENT_ANCHOR 参照）

Step 3: Symbol List 状態確認（新設・必須）
  → kabuStation の登録 symbol リストを取得
  → 対象 symbol が登録済みか確認

Step 4: 条件付き Symbol 再登録
  → Step 3 で symbol が `empty` または対象 symbol が未登録の場合のみ実行
  → 登録済みの場合はスキップ

Step 5: Collector 実行
  → board データ取得
  → Observation Layer へのデータ供給確認
```

**重要制約:**
- Step 3 を省略してはならない（symbol 未確認での collector 実行を禁止）
- Step 4 は条件付きである（必須ではなく、Step 3 の結果に依存）
- 各 Step は前 Step の成功を前提とする（順序固定・スキップ禁止）

### 4.4 再登録が必要な条件

以下のいずれかが真の場合、Symbol 再登録を実行する。

| 条件 | 対応 |
|------|------|
| kabuStation の symbol リストが空 | 全対象 symbol を再登録 |
| 対象 symbol が symbol リストに存在しない | 当該 symbol を登録 |
| 前回登録から再起動が発生した確認あり | symbol リスト状態確認後に判断 |

再起動が発生していない場合でも、**Step 3（状態確認）は必須**とする。

### 4.5 Recovery Sequence と既存制度の関係

本 Sequence は ATOS Operational Protocol（P-20260331-004）の
**Preflight チェック**に組み込む。

具体的には：

```
P-20260331-004 Section 3（Preflight）に以下を追加:
  - kabuStation 再起動後は Restart Recovery Sequence（P-20260403-001 Section 4.3）を実行すること
  - Collector 実行前に symbol 登録状態を確認すること
```

### 4.6 制度的性質

```
Symbol 登録状態の確認は、
Collector 実行の前提条件（Preflight 要件）として制度化される。
未確認での Collector 実行は手順違反とみなす。
```

## 5. Part B — Preview Freshness Calibration

### 5.1 現行パラメータと観測値

| 項目 | 値 |
|------|----|
| 現行 `freshness_limit_ms` | 5,000 ms（5秒） |
| ライブ計測値（観測） | ≈ 25,427 ms（約25秒） |
| 結果 | `snapshot_stale` → READY ブロック |

### 5.2 問題の性質

現行 5,000 ms は `simulated_order` 検証フェーズ向けの保守値である。
ライブデータ環境では、以下の要因により freshness が 5 秒を超えることがある。

| 要因 | 説明 |
|------|------|
| Collector の実行サイクル | board 取得 → DB 書き込み → Preview 参照までの処理時間 |
| WSL ↔ PowerShell ブリッジ遅延 | 境界越えの通信コスト |
| kabuStation の更新頻度 | ブローカー側の板更新間隔 |

これらはいずれも正常な動作であり、5,000 ms 超過はシステム異常を意味しない。

### 5.3 較正方針

**no-override 原則は維持する。**

```
override による freshness 無効化は禁止（P-20260331-001 継承）。
パラメータ値の変更のみで較正する。
```

### 5.4 提案値

| 項目 | 現行値 | 提案値 | 根拠 |
|------|--------|--------|------|
| `freshness_limit_ms` | 5,000 ms | **30,000 ms**（provisional calibrated value） | ライブ計測値（≈25,427ms）の上限余裕を確保。n=1実測に基づく較正値。将来の実測蓄積により再較正可能。 |

**30,000 ms の根拠:**

- 観測値 25,427 ms に対し約 18% のマージンを確保
- P-20260331-002 の TTL（Approval 有効期限 60,000 ms）の半分以下を維持
- P-20260330-001（Hard Limit）の `freshness > 60s → DATA_INVALID` との整合

```
30,000 ms < 60,000 ms（Approval TTL）
30,000 ms < 60,000 ms（DATA_INVALID 境界）
→ 既存の Hard Limit 体系を侵食しない
```

### 5.5 market_closed 時の扱い

`market_closed` は Freshness とは独立した別チェックである。

- 市場閉場時は `market_closed` により READY ブロックが発生する
- これはシステムの正常動作であり、パラメータ変更の対象外とする
- `freshness_limit_ms` の変更は `snapshot_stale` にのみ影響する

```
market_closed → パラメータ変更では解決不可・正常動作として受容
snapshot_stale → freshness_limit_ms 較正で対応可能
```

### 5.6 較正後の Hard Limit 体系

| チェック | 閾値 | 超過時の判定 | 変更有無 |
|---------|------|------------|--------|
| `freshness_limit_ms` | **30,000 ms**（変更） | `snapshot_stale` → NOT_READY | **変更** |
| `DATA_INVALID` 境界 | 60,000 ms | DATA_INVALID | 変更なし |
| Approval TTL | 60,000 ms | Gate 拒否 | 変更なし |
| READY context expiry | 60,000 ms | expired | 変更なし |

### 5.7 将来的な較正方針

`freshness_limit_ms` は固定値ではなく、運用知見に基づき較正可能なパラメータとして扱う。

```
較正変更の条件:
- 実測値の蓄積（n ≥ 5 セッション）に基づく
- Founder 承認を要する
- override による回避は引き続き禁止
```
