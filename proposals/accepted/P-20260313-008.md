# P-20260313-008 -- Execution Governance Framework

## 1. Research Hypothesis

Defining Execution as an independent institutional category with explicit
authorization requirements, mode distinctions, and auditability rules will
prevent unauthorized or untracked live actions in AI Trading OS, completing
the Phase A governance architecture.

## 2. Context Reference

- CRC-01 (Execution Authority Gap / Articles A-D)
- CRC-02 (AI Identity / Articles E-G)
- CRC-05 (Market Observation Governance) [TE:91]
- CRC-06 (Hypothesis Governance Framework) [TE:92]
- constitution.md v1.3 Article A-D

## 3. Part I -- Core Definitions

### 3.1 Execution

Execution is any act that causes an irreversible or quasi-irreversible change
to the external state of the world or to the internal institutional state of ATOS.

Execution includes at minimum:
- External API order submission
- Order cancellation
- Position adjustment
- Live capital movement instruction
- Production environment configuration change
- Execution authority grant or revocation
- Live-environment-affecting scheduling
- Automation job activation
- Emergency stop release

### 3.2 Non-Execution

The following are not Execution in principle:
- Observation collection
- Hypothesis generation
- Proposal drafting
- Deliberation
- Simulation
- Backtesting
- Paper mode pseudo-orders
- Read-only state inspection

Exception: If any of the above is structurally connected to a live action
automatically, it is treated as Execution.

### 3.3 Execution Artifact

Execution Artifact is an institutional object that is the target or basis
of Execution. Examples:

```
execution_request
execution_policy
risk_limit
broker_route
live_strategy_binding
kill_switch_state
approval_record
```

### 3.4 Execution Authority

Execution Authority is the institutional right to approve an Execution as
valid. It is not inherited, auto-elevated, or implied.

### 3.5 Policy / Adapter Boundary

`execution_policy` = institutional layer object
  What is permitted, who can approve, which risk constraints apply.

`broker_adapter` = infrastructure layer object
  Which API to connect to, order format translation, connection failure handling.

CRC-07 defines execution_policy. broker_adapter is outside this scope.

## 4. Part II -- Institutional Principles

```
Principle 1: Deliberation is not execution
  No AI output alone carries execution authority.

Principle 2: Execution requires explicit authorization
  No implicit inheritance, auto-escalation, or tacit consent.

Principle 3: Human accountability remains final
  AI may propose, review, warn, and record. Responsibility does not
  transfer to AI in current phase.

Principle 4: Risk precedes speed
  Institutional integrity and loss prevention take priority over
  execution speed.

Principle 5: Every live action must be reconstructable
  All events related to live execution must be traceable after the fact.

Principle 6: Policy legitimacy is distinct from transport mechanics
  What is institutionally permitted (policy) is separate from how
  it is technically transmitted (adapter).
```

## 5. Part III -- Execution Modes

### 5.1 Simulation Mode

Full simulation. No external impact.
Historical replay, virtual fills, virtual capital only.
Founder individual approval: not required.

### 5.2 Paper Mode

References live market data. No real orders submitted.
Pseudo-execution evaluation. Founder individual approval: not required.

### 5.3 Shadow Mode

Monitors live conditions and generates theoretical execution decisions
without submitting orders.
Used for auditing live execution candidates.
Execution candidates recorded as shadow_execution_candidate trace_events.
Founder individual approval: not required for standard operation.
Note: system-wide automation requires separate definition.

### 5.4 Assisted Live Mode

Real execution with mandatory human explicit approval per execution.
This is the target initial live form for Phase A ATOS.
Minimum requirements: see Rule B (Section 8).

### 5.5 Autonomous Live Mode

Autonomous execution within pre-approved policy boundaries.
This mode is NOT OPEN in CRC-07.
Future introduction requires an independent CRC.

## 6. Part IV -- Role Boundaries

```
Collector (collector_core)
  CAN:  Generate observations
        Attach non-binding execution suggestions
  CANNOT: Execute directly

Proposer (claude_proposer)
  CAN:  Draft execution architecture / policy proposals
        Draft individual execution institutional cases
  CANNOT: Hold execution authority

Librarian (gpt_librarian)
  CAN:  Institutional consistency review
        Precedent organization
        Policy conflict identification
  CANNOT: Hold execution authority

Registrar (claude_registrar)
  CAN:  Record approved institutional documents, update state
  CANNOT: Approve live execution alone

Risk Manager
  CAN:  deny / hold / pass execution_request
        Propose or request emergency stop
  Phase A: human-primary
  AI: risk warning / advisory only, no deny authority in Phase A

Founder / Human Authority
  Final live execution approval authority
  Final emergency stop authority
  Authority delegation definition authority
```

## 7. Part V -- Execution Request Model

### 7.1 execution_request Object

```yaml
execution_request_id: ER-YYYYMMDD-NNN
timestamp: ISO8601
requested_by: actor_id
actor_type: proposer | registrar | founder
mode: simulation | paper | shadow | assisted_live | autonomous_live
target: instrument_id or system component
action: description of the action
rationale: natural language rationale
source_observations: [trace_event_id]
source_hypotheses: [hypothesis_id]
risk_class: low | medium | high | critical
notional_limit: numeric
time_in_force: ISO8601 duration or null
preconditions: []
approval_required_from: [actor_id]
status: drafted | submitted | under_review | approved | denied |
        expired | executed | partially_executed | cancelled |
        emergency_blocked
```

### 7.2 Hypothesis Reference Rule

An execution_request targeting Assisted Live Mode MUST contain at least
one valid source_hypotheses entry.

This preserves the institutional pipeline continuity:
```
Observation -> Hypothesis -> Bet Proposal -> Execution
```

Exception: simulation / paper mode manual test execution.

### 7.3 Approval States

All status transitions must be recorded as trace_events.

```
drafted
submitted
under_review
approved
denied
expired
executed
partially_executed
cancelled
emergency_blocked
```

## 8. Part VI -- Minimum Approval Rules

### Rule A -- Simulation / Paper / Shadow

Founder individual approval not required.
Note: system-wide automation requires separate definition.

### Rule B -- Assisted Live Mode

Minimum requirements:
1. Valid execution_request (including source_hypotheses for live)
2. Risk Manager review (deny/hold/pass)
3. Human final approval (Founder)
4. trace_event recording before and after execution
5. Kill switch availability confirmed

### Rule C -- Autonomous Live Mode

NOT APPROVED in CRC-07.
Future introduction requires an independent CRC.

## 9. Part VII -- Risk Constraints

Execution Governance requires at minimum the following risk objects,
referenced from execution_policy:

```
max_notional_per_order
max_daily_loss
max_position_per_instrument
max_open_orders
trading_session_boundary
instrument_allowlist
mode_allowlist
broker_route_allowlist
```

## 10. Part VIII -- Emergency Powers

### 10.1 Kill Switch

ATOS must have a kill switch capable of stopping at minimum:
- New order submission
- Automatic execution loops
- External order jobs
- Autonomous approval chains

### 10.2 Emergency Stop Authority

Founder / Human Authority holds final stop authority.
Risk Manager may propose or request emergency stop activation.

### 10.3 Emergency Override Logging

All emergency stops, releases, and exceptional approvals must be
recorded as special trace_events.

### 10.4 Kill Switch State Change Logging

All kill switch state changes must be recorded as trace_event:

```json
{
  "event_type": "kill_switch_state_changed",
  "agent_id": "<actor_id>",
  "actor_type": "<actor_type>",
  "metadata": {
    "previous_state": "<state>",
    "new_state": "<state>",
    "reason": "<rationale>",
    "related_execution_policy_id": "<id_or_null>",
    "environment": "<live|paper|shadow|simulation>"
  }
}
```

new_state values:
```
enabled
disabled
partially_restricted
emergency_locked
```

Kill switch release is treated as a high-risk institutional operation.

## 11. Part IX -- Auditability Requirements

All events related to live execution must record:

```
event_id
timestamp
actor_id
actor_type
event_type
execution_request_id
related_policy_id
target
decision
reason
mode
environment
hash / version reference
```

Reconstruction requirements (must be answerable from trace_event):
1. Why was execution performed?
2. Who was involved?
3. Under which policy?
4. Which observations and hypotheses were relied upon?
5. At which point was stopping possible?

### 11.1 Shadow Audit Event

```yaml
event_type: shadow_execution_candidate
description: >
  Records a hypothetical execution candidate generated under Shadow Mode
  without sending a live order.
minimum_fields:
  - timestamp
  - actor_id
  - actor_type
  - hypothesis_id
  - target
  - proposed_action
  - rationale
  - mode
  - decision_context
```

This is not a live order but is a subject of institutional audit.

## 12. Part X -- Constitutional Interpretation

```
AI may:   generate execution candidates
          support execution review
          issue risk warnings

AI may not: hold execution authority unless explicitly granted
            institutionally
```

This principle prevents ATOS from converting AI capability directly
into institutional authority.

Autonomous Live Mode is a future deliberation subject and requires
independent CRC and constitutional amendment if adopted.

## 13. Part XI -- Institutional Decisions (CRC-07)

```
Decision 1: Execution defined as independent institutional category
Decision 2: Live execution requires Risk Manager review + Human final approval
Decision 3: Autonomous Live Mode not open; requires independent CRC
Decision 4: All execution must be reconstructable via execution_request + trace_event
Decision 5: Kill switch and emergency stop logging are mandatory institutional requirements
Decision 6: Shadow Mode execution candidates are institutional audit subjects
            (shadow_execution_candidate event_type)
Decision 7: CRC-07 defines execution_policy only; broker_adapter is out of scope
Decision 8: Assisted Live execution_request must contain minimum 1 source_hypotheses
```

## 14. Part XII -- Reserved Future Constitutional Agenda

If Autonomous Live Mode is considered in future:

```
Article H (candidate): Autonomous Execution Conditions
  - Permitted mode definition
  - Required preconditions (backtest period, win rate threshold, etc.)
  - Auto-stop conditions (daily loss limit breach, etc.)

Article I (candidate): Execution Override Rights
  - Founder immediate stop authority
  - AI auto-stop obligation

Article J (candidate): Accountability Chain for Autonomous Execution
  - Responsibility under autonomous execution
  - Recursive stop authority on exception
```

This section is a reserved agenda, not an adoption request.

## 15. Future Expansion Path (Phase B+ candidates)

```
- AI-assisted Risk Manager expansion (requires independent CRC)
- strategy-level authorization (requires independent CRC)
- Shadow Mode integration with confidence calibration (CRC-06 E4)
- execution policy versioning + runtime deployment approval
- Autonomous Live Mode (requires independent CRC + constitution amendment)
- policy mandatory cool-down period on change
- order-level vs. strategy-level authorization separation
```

## 16. Confidence Statement

High. This proposal is the direct institutional enactment of CRC-07
deliberation consensus. All definitions, rules, and constraints are
consistent with prior CRC outcomes and the constitution v1.3.

## 17. Limitations

- Risk Manager is Founder-only in Phase A. AI risk advisory is non-binding.
- Broker adapter definition is deferred to a separate operational proposal.
- Autonomous Live Mode is not available. Any attempt requires independent CRC.
- kill switch technical implementation is outside this governance proposal.
- execution_request status automation (expiry detection etc.) is operationally
  outside this governance proposal.
