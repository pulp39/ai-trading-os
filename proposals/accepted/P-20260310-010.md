# P-20260310-010

## Title

Founder記録ディレクトリの制度化

## Status

accepted

## Proposal Type

governance / institutional_structure

---

## Scope Statement

本提案は、Founderが発行する制度記録を格納・管理するためのディレクトリ構造とその制度的位置づけを定義する。

Founder記録はProposalではない。Founderが制度的意思決定者として直接発行する記録であり、提案プロセスを経ない。本提案はこのFounder記録を制度的トレーサビリティの枠組みに接続するための構造を定義する。

---

## Purpose

P-20260310-009の承認およびFR-20260310-001の発行により、Founder記録という新たな記録種別が制度上に出現した。しかし現時点では、Founder記録を格納する制度的な場所が存在しない。

Founderの制度的意思決定は制度記録として保全されなければならない。これは憲法第2条（Traceability原則）および第8条（Traceable Knowledge不変原則）の要請でもある。

本提案はFounder記録ディレクトリを制度化し、Founder記録の格納・編集権限・trace_event対応づけの手続きを定義する。

---

## Part I — ディレクトリ構造
```
repository/
  proposals/
    draft/
    accepted/
  founder_records/
    proposals/
    approvals/
    directives/
    constitutional/
```

### ディレクトリ用途定義

| ディレクトリ | 格納対象 |
|---|---|
| `founder_records/proposals/` | Founderが直接起草・発行した制度提案記録 |
| `founder_records/approvals/` | Founderによる提案承認記録 |
| `founder_records/directives/` | Founderが発行した制度的指示・通達 |
| `founder_records/constitutional/` | 憲法改正・制度変更に関するFounder記録 |

---

## Part II — Founder記録の制度的位置づけ

### 2.1 Founder記録はProposalではない

Founder記録はProposal registryに登録されない。Founder記録はFounderが制度的意思決定者として直接発行するものであり、提案・審査・承認のプロセスを経ない。

### 2.2 制度的効力

Founder記録の制度的効力は、格納先ディレクトリの存在に依存しない。Founderが発行した時点で制度的効力を持つ。格納先ディレクトリはトレーサビリティのための構造であり、効力の源泉ではない。

### 2.3 trace_eventとの接続

各Founder記録はtrace_eventに接続される。trace_event_idはLibrarianがFounder記録のヘッダーに追記する。これはLibrarian権限での制度的対応づけとして扱う。

---

## Part III — Founder記録のID体系とヘッダー

### 3.1 ID形式
```
FR-YYYYMMDD-NNN
```

例: FR-20260310-001

### 3.2 標準ヘッダーフィールド
```yaml
founder_record_id: FR-YYYYMMDD-NNN
record_type: approval | proposal | directive | constitutional
date: YYYY-MM-DD
original_date: YYYY-MM-DD  # 遡及記録の場合のみ
retroactive: true | false
related_proposal_id: P-YYYYMMDD-NNN  # 関連する場合
trace_event_id: [Librarian追記]
author: Founder
edit_authority: Founder only
```

---

## Part IV — 編集権限

### 4.1 編集権限の原則

Founder記録の編集権限はFounderのみが持つ。これはCODEOWNERSおよびブランチ保護設定により技術的に強制される。

### 4.2 例外

Librarianは登録後にtrace_event_idをFounder記録ヘッダーに追記する操作のみ実施できる。この操作はFounder記録の内容を変更しない制度的対応づけとして扱う。

---

## Part V — trace_event形式

Founder記録の登録に対応するtrace_eventは以下の形式を用いる。
```json
{
  "event_type": "founder_record_created",
  "metadata": {
    "founder_record_id": "FR-YYYYMMDD-NNN",
    "record_type": "approval | proposal | directive | constitutional",
    "file_path": "founder_records/[subdirectory]/FR-YYYYMMDD-NNN.md",
    "related_proposal_id": "P-YYYYMMDD-NNN"
  }
}
```

## Part VII — FR-20260310-002の内容定義

本提案の記録範囲として、FR-20260310-002の内容を以下に定義する。
```yaml
founder_record_id: FR-20260310-002
record_type: proposal
date: 2026-03-11  # P-20260310-011起草日
original_date: 2026-03-10  # Founderが一時招待提案を発した日
retroactive: true
related_proposal_id: P-20260310-011
trace_event_id: [Librarian追記]
author: Founder
edit_authority: Founder only
```

**内容:**

Founderは2026-03-10に、OpenClawをAI Collector候補として制度審査に一時招待することを提案した。この提案は口頭（テキスト）による制度的意思表明として記録される。

提案の主旨: OpenClawをP-20260310-009のAI Collector候補審査フレームワークの最初の審査対象として招待し、制度の強度を試すこと。

この記録はP-20260310-011の制度的先行記録として機能する。
