# P-20260313-005 — Market Observation Governance

## 1. Research Hypothesis

Defining a canonical Instrument Model, Observation Model, and
Observation Integrity Model for Phase A will establish the
foundational data contract between the Observation Layer and
the AI Hypothesis Layer, enabling traceable, integrity-governed
market observation.

## 3. Part I — Instrument Model

### 3.1 Definition

An Instrument is a market-level institutional object within
AI Trading OS. It represents a tradeable entity as defined
by the market, independent of any specific data source or
API implementation.

An Instrument is NOT a source-level identifier. Source-specific
identifiers (e.g., kabuStation Symbol and Exchange codes) are
handled in the Mapping Layer and do not define the canonical object.

### 3.2 Instrument Structure (Phase A Minimum)

```
instrument_id    : Canonical identifier within AI Trading OS
                   Format: {asset_class}_{symbol_canonical}
                   Example: equity_7203, equity_9433
asset_class      : equity | futures | options
symbol_canonical : Human-readable identifier
                   Example: "7203" (Toyota), "9433" (KDDI)
market           : Market classification
                   Example: TSE_prime, OSE
```

### 3.3 Mapping Layer

The Mapping Layer translates between Instrument (market-level)
and source-specific API identifiers. This layer is the
responsibility of the Collector.

```
Mapping Layer fields (source-specific, not institutional):
  source        : "kabuStation"
  api_symbol    : Raw API symbol value (e.g., "7203")
  api_exchange  : Raw API exchange code (e.g., 1)
  -> resolves to -> instrument_id: "equity_7203"
```

The Mapping Layer does not constitute an institutional record.
It is an operational translation function within the Collector.

### 3.4 Phase A Scope

In Phase A, instrument coverage is limited to equities
observable via kabuStation API. Futures and options instruments
are recognized as valid asset_class values but are not actively
observed in Phase A. Expansion requires a separate proposal.

## 4. Part II — Observation Model

### 4.1 Canonical Observation Type

```
observation_type : price_bar
interval         : 1m
unit             : yen (JPY)
```

The canonical observation in Phase A is a 1-minute price bar
for a given instrument.

### 4.2 Observation Report (Collector Output)

The Collector generates observation reports per the provisional
schema defined in P-20260313-004 section 4.3. This proposal
confirms that schema and adds the following required fields
for Instrument Model compliance:

```yaml
observation_type: price_bar
interval: 1m
source: kabuStation
collector_id: collector_core
phase: Phase A
instrument_id: equity_<symbol>        # canonical instrument
api_symbol: <kabuStation symbol>      # mapping layer
api_exchange: <kabuStation exchange>  # mapping layer
timestamp: <ISO8601>
data:
  open: <value>
  high: <value>
  low: <value>
  close: <value>
  volume: <value>
analysis:
  summary: <optional>
  hypothesis: <optional, non-binding, flagged>
```

### 4.3 trace_event Structure (canonical_observation)

```json
{
  "event_type": "canonical_observation",
  "agent_id": "collector_core",
  "actor_type": "collector",
  "content": "<one-line summary>",
  "metadata": {
    "instrument_id": "equity_7203",
    "observation_type": "price_bar",
    "interval": "1m",
    "source": "kabuStation",
    "api_symbol": "7203",
    "api_exchange": 1,
    "collector_id": "collector_core",
    "phase": "Phase A",
    "quality_tier": "canonical",
    "integrity_role": "canonical",
    "timestamp": "<ISO8601>"
  }
}
```

## 5. Part III — Observation Integrity Model

### 5.1 Institutional Rules

**Rule 1**
Observation integrity is governed by source policy before
comparison policy.

**Rule 2**
In Phase A, kabuStation is the designated canonical source
for equity price_bar observations.

**Rule 3**
Observations from non-canonical sources may exist as
supplemental observations but do not override canonical
observations automatically.
In Phase A, supplemental observations are not recorded in
trace_event. This constraint may be lifted by a future
proposal.

**Rule 4**
A conflict exists when two observations refer to the same
instrument_id, timestamp, and observation_type but differ
in payload.

**Rule 5**
Conflict resolution is deferred beyond Phase A unless
explicitly authorized by Founder and enacted through proposal.

**Rule 6**
In Phase A, all trace_events of event_type canonical_observation
carry integrity_role = "canonical".
No other integrity_role value is assigned in Phase A.

### 5.2 Integrity Role Vocabulary

```
canonical    : Current governance policy designates this as
               the authoritative observation. (Phase A: only value)
supplemental : Auxiliary observation from non-canonical source.
               (Defined; not recorded in Phase A trace_event)
conflict     : Observation in conflict with canonical value.
               (Defined; resolution deferred to Phase B+)
```

### 5.3 Single-Source Canonical Regime (Phase A)

```
Phase A Integrity Regime: single-source canonical
  canonical source     : kabuStation
  canonical type       : price_bar
  canonical interval   : 1m
  integrity_role       : canonical (all Phase A observations)
  supplemental         : concept defined, not operationalized
  conflict resolution  : deferred
```

## 6. Part IV — Institutional Data Pipeline

As of P-20260313-005, the institutional data pipeline is
formally defined:

```
Market
  |
  v
Collector (OpenClaw / collector_core)
  | passive retrieval via kabuStation API
  v
Mapping Layer (Collector-internal)
  | api_symbol + api_exchange -> instrument_id
  v
Canonical Observation (observation report)
  | YAML format per section 4.2
  v
trace_event (canonical_observation)
  | Registrar records; integrity_role = canonical
  v
AI Hypothesis Layer (Proposer)
  | interprets canonical_observations
  v
Hypothesis / Proposal
  |
  v
[Execution — outside current scope]
```

## 7. Future Expansion Path

```
Phase B candidates (require separate proposals):
  - Multi-source observation (supplemental activation)
  - Conflict detection and resolution engine
  - Futures / options instrument expansion
  - observation_type expansion beyond price_bar
  - instrument_id registry formalization
  - Observation Rate Governance (noted in AG-20260313-002)
```

## 8. Confidence Statement

High. This proposal is the direct institutional enactment of
CRC-05 deliberation consensus. All three models (Instrument,
Observation, Integrity) are internally consistent and traceable
to the deliberation record.

## 9. Limitations

- Instrument naming convention (instrument_id format) is
  provisional for Phase A. Formal registry and naming policy
  may require a future proposal.
- The Mapping Layer definition is operational; Collector
  implementation is a separate technical matter.
- price_bar OHLCV field availability depends on kabuStation
  API response. If only close/volume are available, a schema
  clarification proposal may be needed.
- Phase A covers equity only. Futures/options require a
  separate proposal.
