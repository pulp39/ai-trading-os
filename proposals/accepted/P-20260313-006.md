# P-20260313-006 -- Hypothesis Governance Framework

## 1. Research Hypothesis

Defining a canonical Hypothesis Object, Bet Proposal Object, and
Hypothesis Evaluation Framework for Phase A will establish the
AI reasoning layer of AI Trading OS, enabling traceable, governed
prediction generation and systematic evaluation of AI judgment.

## 3. Part I -- Hypothesis Object

### 3.1 Definition

A Hypothesis is an AI-generated market interpretation recorded as
an institutional object. It represents the Proposer's structured
prediction about a specific instrument, grounded in canonical
observations.

```
Hypothesis = market interpretation
Hypothesis != Order
Hypothesis != Position
Hypothesis != Execution instruction
```

### 3.2 Hypothesis Object Structure

```
hypothesis_id      : UUID (canonical identifier)
instrument_id      : CRC-05 canonical object (e.g., equity_7203)
hypothesis_type    : trend | mean_reversion | breakout |
                     event_driven | other
                     (Phase A: free string also permitted)
direction          : long | short | neutral
confidence         : 0.0 - 1.0 (Phase A: AI self-assessment)
confidence_basis   : Natural language rationale (required)
evidence           : JSON structured rationale
                     e.g. {"indicator": "SMA_cross", "fast": 5, "slow": 20}
source_observation : {trace_event_id, instrument_id, timestamp}
proposer_id        : e.g., claude_proposer
generated_at       : ISO8601
expiry_timestamp   : ISO8601 (required -- set by Proposer)
status             : generated | active | expired | invalidated
invalidated_by     : trace_event_id (only when invalidated)
phase              : Phase A
```

### 3.3 Hypothesis Lifecycle

```
generated
  Hypothesis recorded in trace_event.
  Not yet confirmed as active.

active
  Valid hypothesis. May serve as basis for Bet Proposal.

expired
  expiry_timestamp passed.
  Automatically loses institutional effect.
  Eligible for evaluation.

invalidated
  Contradicting observation or evidence explicitly invalidates.
  invalidated_by records the responsible trace_event_id.
  Eligible for evaluation.
```

All lifecycle transitions are recorded as trace_events.
Hypotheses are never deleted.

### 3.4 Hypothesis Generation Trigger

Hypothesis generation may be triggered by:
- New canonical_observation (Observation-triggered, recommended)
- Autonomous AI periodic reasoning (permitted)

Both modes are institutionally valid in Phase A.
All generated hypotheses are recorded in trace_event regardless of trigger.

### 3.5 Confidence Model (Phase A)

```
confidence         = AI self-assessment (0.0 - 1.0)
confidence_basis   = required natural language explanation
```

Phase A confidence data constitutes the training dataset for
future statistical confidence calibration (Phase B+).

```
Phase B+ path:
  confidence_v2 = statistical model output
  (backtest win rate, Sharpe, model score, etc.)
```

### 3.6 Hypothesis trace_event Structure

```json
{
  "event_type": "hypothesis_generated",
  "agent_id": "claude_proposer",
  "actor_type": "proposer",
  "content": "<one-line summary>",
  "metadata": {
    "hypothesis_id": "<uuid>",
    "instrument_id": "equity_7203",
    "hypothesis_type": "trend",
    "direction": "long",
    "confidence": 0.72,
    "confidence_basis": "<rationale>",
    "expiry_timestamp": "<ISO8601>",
    "status": "generated",
    "source_observation_id": "<trace_event_id>",
    "phase": "Phase A"
  }
}
```

Additional event_types for lifecycle transitions:
```
hypothesis_updated
hypothesis_invalidated
```

## 4. Part II -- Bet Proposal Object

### 4.1 Definition

A Bet Proposal is an AI-generated conditional execution intent.
It represents the Proposer's structured recommendation for market
action, combining a Hypothesis with explicit risk parameters.

```
Bet Proposal
= Hypothesis
+ Risk Parameters
+ Execution Conditions

Bet Proposal = Conditional Execution Intent (advisory)
Bet Proposal != Execution Order
Bet Proposal != Position
```

### 4.2 Bet Proposal Object Structure

```
bet_id              : UUID
hypothesis_id       : Required reference to active Hypothesis
instrument_id       : CRC-05 canonical object
direction           : long | short | neutral

-- Risk Parameters --
position_size_type  : fixed_units (Phase A only)
                      pct_capital | risk_based (reserved, Phase B+)
position_size_value : Numeric value in units of position_size_type

entry_condition     : "market" | "limit:<price>" | "on_open"
stop_loss           : price | null
  (if null, stop_loss_rationale field required)
take_profit         : price | null
max_holding_period  : ISO8601 duration | null
                      e.g., "P5D" (5 business days)

-- Metadata --
proposer_id         : e.g., claude_proposer
generated_at        : ISO8601
phase               : Phase A
bet_status          : proposed | accepted | rejected | executed | expired
rejected_reason     : Natural language (only when rejected)
```

### 4.3 Bet Proposal Generation Trigger (Phase A)

Bet Proposal generation requires:
1. A valid active Hypothesis exists (hypothesis_id binding)
2. Generation is permitted by Proposer autonomous judgment
   (Founder trigger recommended in Phase A)

All generated Bet Proposals are recorded as trace_events.

### 4.4 Bet Proposal Institutional Rules

**Rule B1 -- Advisory Only**
Bet Proposals do not trigger execution.
A Bet Proposal is an advisory research artifact.
Constitution Article 3 applies.

**Rule B2 -- Hypothesis Binding**
Every Bet Proposal MUST reference a valid hypothesis_id.
A Bet Proposal without hypothesis_id is institutionally invalid.

**Rule B3 -- Risk Parameter Completeness**
position_size_type and position_size_value MUST be present.
stop_loss may be null but requires stop_loss_rationale field.

**Rule B4 -- trace_event Recording**
All Bet Proposal lifecycle transitions are recorded as trace_events:
```
bet_proposed
bet_accepted
bet_rejected
bet_expired
```

**Rule B5 -- Execution Authority Separation**
Only Founder authority may change bet_status to accepted.
AI agents may not move a Bet Proposal to accepted status.
Constitution Article A and Article D apply.

**Rule B6 -- Immutability of Proposal Content**
Risk parameters of a proposed Bet Proposal cannot be modified.
A new Bet Proposal must be generated instead.
This preserves the integrity of the institutional record.

### 4.5 Bet Proposal trace_event Structure

```json
{
  "event_type": "bet_proposed",
  "agent_id": "claude_proposer",
  "actor_type": "proposer",
  "content": "<one-line summary>",
  "metadata": {
    "bet_id": "<uuid>",
    "hypothesis_id": "<uuid>",
    "instrument_id": "equity_7203",
    "direction": "long",
    "position_size_type": "fixed_units",
    "position_size_value": 100,
    "entry_condition": "limit:3400",
    "stop_loss": 3200,
    "take_profit": 3600,
    "max_holding_period": "P5D",
    "bet_status": "proposed",
    "phase": "Phase A"
  }
}
```

## 5. Part III -- Hypothesis Evaluation Framework

### 5.1 Definition

Hypothesis Evaluation is the institutional process of comparing
a Hypothesis prediction against canonical market observations
after the hypothesis lifecycle ends.

Evaluation is triggered by:
- Hypothesis expiry (expired status)
- Hypothesis invalidation (invalidated status)
- Manual Founder or Registrar action

### 5.2 Hypothesis Evaluation Object Structure

```
evaluation_id        : UUID
hypothesis_id        : Evaluated hypothesis reference
instrument_id        : CRC-05 canonical object
evaluation_timestamp : ISO8601
evaluation_trigger   : "expired" | "invalidated" | "manual"

-- Outcome --
outcome              : hit | miss | partial | inconclusive
  hit          = direction correct + confidence threshold met
  miss         = direction incorrect
  partial      = direction correct but confidence miscalibrated
  inconclusive = insufficient data, market closed, etc.

direction_correct    : true | false | null (inconclusive)
entry_price_ref      : Price at hypothesis generation (from observation)
exit_price_ref       : Price at expiry/invalidation (from observation)
price_change_pct     : (exit - entry) / entry

-- Confidence Calibration --
stated_confidence    : Hypothesis confidence at generation
outcome_binary       : 1 (hit) | 0 (miss) | null
calibration_note     : Natural language (optional)

-- Source --
evaluated_by         : "registrar" | "founder"
                       ("system" reserved for Phase B+ automation)
source_observations  : [trace_event_id] (observations used for evaluation)
phase                : Phase A
```

### 5.3 Hypothesis Evaluation Institutional Rules

**Rule E1 -- Non-Punitive Evaluation (Phase A)**
Evaluation is for recording and learning purposes only.
Evaluation results do not affect Proposer's institutional privileges
or Bet Proposal generation authority in Phase A.

**Rule E2 -- Evaluation Requires Observation**
Evaluation requires the existence of canonical_observations
covering the hypothesis period.
If observations are unavailable, outcome = inconclusive.

**Rule E3 -- Evaluation Immutability**
Once recorded, evaluation records are not modified.
Disputes require a separate evaluation_dispute record.

**Rule E4 -- Aggregate Metrics Computed, Not Stored**
Derived metrics (hit_rate, calibration_score, Sharpe, etc.) are not
stored in trace_event.
They are computed from trace_event records when needed.
This preserves trace_event as immutable atomic events.

**Rule E5 -- Evaluation Independence**
Evaluation does not modify the target Hypothesis object.
Hypothesis lifecycle management is independent.

### 5.4 Integrity Role Vocabulary

```
evaluated_by values (Phase A):
  registrar  = Registrar-executed evaluation
  founder    = Founder-executed evaluation
  system     = reserved (Phase B+ automated evaluation engine)
```

### 5.5 Evaluation trace_event Structure

```json
{
  "event_type": "hypothesis_evaluated",
  "agent_id": "claude_registrar",
  "actor_type": "registrar",
  "content": "Hypothesis <id> evaluated: <outcome>",
  "metadata": {
    "evaluation_id": "<uuid>",
    "hypothesis_id": "<uuid>",
    "instrument_id": "equity_7203",
    "outcome": "hit",
    "direction_correct": true,
    "stated_confidence": 0.72,
    "price_change_pct": 0.021,
    "evaluation_trigger": "expired",
    "evaluated_by": "registrar",
    "phase": "Phase A"
  }
}
```

## 6. Part IV -- Hypothesis Layer Architecture

As of P-20260313-006, the Hypothesis Layer is formally defined:

```
[Observation Layer -- CRC-05]
  canonical_observation (trace_event)
  source: kabuStation / integrity_role: canonical

        |
        v

[Hypothesis Layer -- CRC-06]

  hypothesis_generated  (trace_event)
    status: generated -> active
    expiry_timestamp: set by Proposer

        |
        v  (Proposer analysis)

  bet_proposed  (trace_event)
    status: proposed
    position_size_type: fixed_units (Phase A)

        |
        v  (Founder review)

  bet_accepted / bet_rejected  (trace_event)
    [accepted = ready for Execution Layer]
    [rejected = lifecycle ends]

        |
        v  (post-lifecycle)

  hypothesis_evaluated  (trace_event)
    outcome: hit | miss | partial | inconclusive
    confidence calibration data recorded

[Execution Layer -- outside CRC-06 scope]
  Trading VM / Founder authority only
```

### 6.1 Full Institutional Pipeline

```
Market
  |
  v
Collector (OpenClaw / collector_core)
  | kabuStation API
  v
Mapping Layer
  | api_symbol -> instrument_id
  v
Canonical Observation
  | trace_event: canonical_observation
  v
Proposer (AI Reasoning)
  | trace_event: hypothesis_generated
  v
Bet Proposal
  | trace_event: bet_proposed
  v
Founder Review
  | trace_event: bet_accepted / bet_rejected
  v
Execution Layer (Trading VM)
  |
  v
Hypothesis Evaluation
  | trace_event: hypothesis_evaluated
  v
Institutional Learning Record
```

## 10. Limitations

- Hypothesis evaluation requires canonical_observations for the
  hypothesis period. If observation data is unavailable, outcome
  defaults to inconclusive.
- position_size_type is fixed_units only in Phase A. Portfolio-level
  sizing requires Execution Layer integration.
- Automatic lifecycle management (expiry detection, evaluation
  triggering) is operationally outside this governance proposal.
- Bet Proposal acceptance by Founder is a manual process in Phase A.
