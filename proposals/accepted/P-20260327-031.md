# Proposal: Authorization Lifecycle と Freshness Gate の制度的定義

## Background

P-20260326-030 は `execution_ready`（状態・capability）と `authorization_granted`（意思・authority）の制度的区別を定義した。しかしその定義はauthorizationの「発行」と「消費」の二点に限定されており、その間に存在するライフサイクル——無効化・再利用拒否・鮮度ゲート・境界閉鎖——については明示的な制度定義が存在しなかった。

2026-03-27 に実施されたPhase 6 Authorization Lifecycle Validation において、以下のシーケンスが制度的に検証された：

```
execution_ready
→ authorization_granted
→ authorization_invalidated
→ authorization_reuse_rejected
→ freshness_gate_blocked
→ boundary_closure_confirmed
```

すべてのステップで `authorization_consumed: false`、`external_execution_entered: false`、`boundary_preserved: true` が維持された。

本Proposalはこの検証結果を受け、authorizationのライフサイクル全体と、freshness gateの制度的要件を定義する。

## Problem Statement

P-20260326-030 の定義だけでは以下が未定義のままとなる。

1. `authorization_granted` が「消費されずに終了する」経路の制度的定義がない
2. authorizationの無効化条件と、無効化後の再利用禁止が明文化されていない
3. `execution_ready` の「鮮度」要件が定義されておらず、過去の観測を再利用できるかどうかが曖昧
4. システムが「実行不可能な閉鎖状態」に入る条件と、そこからの再起動経路が未定義

## Objective

1. Authorization Lifecycle を granted / invalidated / consumed の三状態と遷移条件として定義する
2. Freshness Gate を、`execution_ready` 再構成の必須条件として制度化する
3. Boundary Closure 状態とその再起動経路を定義する
4. P-20260326-030 との整合性を保ちながら、定義を拡張する

## Scope

本Proposalは以下を対象とする。

- `authorization_granted` のライフサイクル定義（granted / invalidated / consumed）
- `authorization_invalidated` の発生条件
- `authorization_reuse_rejected` の制度的根拠
- `freshness_gate` の定義と、`execution_ready` 再構成の要件
- `boundary_closure` 状態と `system_state: closed_until_fresh_observation` の定義
- 再起動（reactivation）経路の明示

本Proposalは実行プロトコルの変更を含まない。状態の意味論と遷移条件の定義に限定する。

---

## 1. Authorization Lifecycle の定義

authorizationは以下の**三状態**を持つ制度的オブジェクトとして定義する。

| 状態 | 意味 |
|------|------|
| `granted` | Founderが発行し、有効な状態。意思の記録として完結している |
| `invalidated` | 明示的に無効化された状態。消費されていない |
| `consumed` | 外部送信に使用され、消費された状態 |

`granted` 状態のauthorizationは、`execution_ready` の成立とは独立した「意思」として存在する。両者が揃っていることは実行の前提条件であるが、authorizationの状態が変化するわけではない。

### 遷移条件

```
[granted]
  → invalidated   : Founderが明示的に無効化したとき、またはセッション境界を越えたとき
  → consumed      : external_order_allowed を経て外部送信が完了したとき（Founderの権限下）

[invalidated]
  → （終端）再利用不可。新たな authorization_granted を必要とする

[consumed]
  → （終端）再利用不可。新たな authorization_granted を必要とする
```

### 重要な原則

> `invalidated` と `consumed` はいずれも終端状態であり、同一のauthorizationオブジェクトは再利用できない。
> authorizationは「使い切る」か「捨てる」かしかない。再利用という概念は制度上存在しない。

---

## 2. Authorization Reuse Prohibition

無効化または消費されたauthorizationの再利用は制度上禁止される。

再利用の試みは `authorization_reuse_rejected` として記録され、実行は進まない。

これはP-20260326-030が定めた「authorization_grantedはFounderのみが発行できる」原則の自然な帰結である。新たな実行サイクルには、新たな `authorization_granted` が必要となる。

---

## 3. Freshness Gate の定義

### 鮮度要件

`execution_ready` の再構成には、**新規の正規観測（fresh canonical observation）** が必要である。

過去のセッションまたは過去のタイムスタンプの観測は、`execution_ready` の構成要素として使用できない。

### 鮮度の構成要件

以下が**すべて**成立したとき、observationは「fresh」と見なされる。

- 同一セッション内で取得されたものである
- かつ、直近の市場状態を反映している
- かつ、Collectorによって `trace_event` として記録されている
- かつ、`execution_ready` の根拠として過去に使用されていない（再利用禁止）

### freshness_gate_blocked の発生条件

以下のいずれかが該当するとき、`freshness_gate_blocked` が発生し `execution_ready` の再構成は阻止される。

- 利用可能な観測が存在しない
- 利用可能な観測がすべて過去セッションのものである
- 観測の正規性（trace_event記録）が確認できない

---

## 4. Boundary Closure と Reactivation

### Boundary Closure

以下の条件がすべて成立したとき、システムは `boundary_closure_confirmed` に入る。

```yaml
active_authorization: false
reconstructable_execution_ready: false
external_execution_forbidden: true
```

この状態を `system_state: closed_until_fresh_observation` と定義する。

閉鎖状態は「失敗」ではなく、「安全な終端」である。システムが自らの実行不可能性を正確に認識し、外部送信を禁じた状態を指す。

### Reactivation 経路

閉鎖状態からの再起動には以下のシーケンスが必要である。

```
fresh canonical observation（新規trace_event記録）
→ market_data_ready の再成立
→ market_ready の再成立
→ execution_ready の再構成
→ 新たな authorization_granted（Founderによる発行）
```

このシーケンスはP-20260326-030が定めた状態階層と完全に整合する。

---

## 5. P-20260326-030 との関係

本Proposalは P-20260326-030 を**上書きせず、拡張する**。

| P-20260326-030 が定義したもの | 本Proposalが追加するもの |
|-------------------------------|--------------------------|
| execution_ready の定義 | freshness_gate による再構成条件 |
| authorization_granted の定義 | ライフサイクル（3状態・遷移条件） |
| 両者の必要条件・十分条件の関係 | invalidation / reuse prohibition / closure |
| Proposerの関与範囲 | 変更なし |

---

## 6. 非目標

本Proposalは以下を意図しない。

- authorizationの発行権限の変更（Founderのみが発行できる原則は維持）
- freshness の定量的な有効期限の定義（これは別途検討事項とする）
- 実行プロトコルの変更
- 新しい役割の追加

---

## Conclusion

Phase 6 バリデーションは、AI Trading OSがauthorizationを消費せずに安全に管理・無効化・閉鎖できることを実証した。本Proposalはその検証結果を制度的な定義として明文化し、P-20260326-030が確立した概念体系を完成させるものである。

authorization lifecycleとfreshness gateの定義により、「何が揃えば実行できるか」だけでなく、「何が失われれば実行できなくなるか」が制度として明確になる。

本Proposalは実行を求めるものではなく、制度的定義の拡張として、Librarianによる審議とFounderによる承認を求める。
