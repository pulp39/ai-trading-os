# Proposal P-20260321-021
**Title:** Execution Gate Operationalization
**Date:** 2026-03-21
**From:** Proposer (Claude)
**Reviewed by:** Librarian (GPT)
**Type:** Governance + Technical Implementation Definition
**Status:** Rev.0 — Pending Founder Approval
**Constitutional Impact:** Low（P-020の制度定義を操作可能な形に具体化）
**Scope:** execution_gateをRegistrarタスクとして操作可能にする。実際のgate通過（execution_gate_passed）は対象外。

---

## 1. 背景

P-20260321-020（trace_event: 272）により `execution_gate` が制度的に定義された。

しかし現状、gateは：

```
定義された概念 → 操作できない
```

P-021の目的は、execution_gateを**実際に操作可能なRegistrarタスク**として具体化することである。

```
P-020 → 定義
P-021 → 操作可能化   ← 本 Proposal
P-022 → 実行（gate通過）
```

---

## 2. 目的

- `task_type: execution_gate` を定義する
- preconditionの2層構造（宣言層・実装層）を確立する
- trace_event挿入構造を明示する
- Gate固有のInvariantを追加確定する

---

## 3. Conceptual Layer

### 3.1 Gateは「操作可能な制度行為」

execution_gateは単なる技術的チェックポイントではなく、Registrarによる**意思決定の記録行為**である。

```
Gateは「判断」であり「動作」ではない
```

したがってGateはRegistrarが明示的に「通過させる」行為として設計される。

### 3.2 本 Proposal のスコープ

```
✅ task_type: execution_gate の定義
✅ precondition 2層構造の確立
✅ trace_event 挿入構造の明示
✅ Gate固有 Invariant の追加（I-6〜I-9）
❌ 実際の execution_gate_passed 発行（次Proposal）
❌ real_order 実装（対象外）
```

---

## 4. Execution Layer

### 4.1 task_type: execution_gate（独立定義）

execution_gate は **独立した task_type** として定義する。preview タスクへの統合は行わない。

理由：Gate単一性の原則（I-5）を実装レベルでも維持するため。Gateをpreviewに統合すると `preview → gate（暗黙）` となり、Gateが消滅する。

#### execution_gate_task.json 構造

```json
{
  "task_type": "execution_gate",
  "date": "<ISO8601>",
  "authorized_by": "claude_registrar",
  "linked_preview_event_id": "<order_preview_recorded の trace_event id>",
  "scope": "bounded",
  "preconditions": {
    "requires_preview": true,
    "requires_determinism": true,
    "requires_role_boundary_check": true,
    "requires_scope_clarity": true
  },
  "metadata": {
    "gate_purpose": "<Registrarによる判断根拠の自由記述>"
  }
}
```

### 4.2 Precondition 2層構造

#### 層① 宣言層（task artifact）

`execution_gate_task.json` の `preconditions` フィールドで意図を宣言する。

目的：
- 制度的意図の可読性確保
- 監査時の参照点
- 将来的な自動検証への対応基盤

#### 層② 実装層（bounded script）

スクリプトが実際に検証するロジック。

```
precondition checklist（スクリプト実装）:

① order_preview_recorded が research.trace_event に存在するか
   └─ linked_preview_event_id で検索
② preview結果が deterministic か（同一入力で同一出力が再現可能か）
③ role boundary violation が存在しないか
④ execution scope が明確か（task artifact に明示されているか）
```

いずれか1つでも失敗した場合、スクリプトは停止し `execution_blocked` を記録する。

#### 2層の関係

```
宣言層（task）  → 「何を確認すべきか」の制度的宣言
実装層（script）→ 「実際に確認する」実行安全性の保証

どちらか一方では不十分。両層が揃って初めてGateは機能する。
```

### 4.3 trace_event 挿入構造

gate通過時（P-022以降）に挿入する `execution_gate_passed` の構造（P-020 Section 6.1の再確認・詳細化）：

```json
{
  "event_type": "execution_gate_passed",
  "agent_id": "claude_registrar",
  "summary": "Execution gate passed for real_order",
  "metadata": {
    "linked_preview_event_id": "<order_preview_recorded の id>",
    "gate_task_ref": "<execution_gate_task.json のパス>",
    "authorized_by": "claude_registrar",
    "gate_passed_at": "<ISO8601>",
    "preconditions_verified": true
  }
}
```

> 本 Proposal はこの構造を定義する。実際の挿入は P-022 の対象。

## 5. Governance Layer

### 5.1 Gate固有 Invariant（P-020 I-1〜I-5 への追加）

| # | Invariant | 説明 |
|---|-----------|------|
| I-6 | **One Gate per Preview** | 1つの preview に対して最大1つの gate のみ存在できる |
| I-7 | **Gate is reproducible** | 同じ `preview_event_id` に対して同じgate判断が再生成可能でなければならない |
| I-8 | **Gate is irreversible** | `execution_gate_passed` は取り消せない。無効化は別event（`execution_invalidated`、将来定義）で表現する |
| I-9 | **Gate does not trigger execution** | Gateを通過してもExecutionは自動発生しない。ExecutionはGate通過後に改めて明示的に指示される |

### 5.2 I-6（One Gate per Preview）の意味

複数のGateが存在すると：

```
どれが実行根拠か不明 → executionの唯一性が崩壊
```

スクリプトは実行前に、対象 `preview_event_id` に対してすでに `execution_gate_passed` が存在しないことを確認しなければならない。

### 5.3 I-8（Gate is irreversible）の意味

trace_eventは履歴であり、修正・削除は行わない。Gate判断を取り消す必要が生じた場合は：

```
execution_invalidated（将来定義）
```

という別eventで表現する。これにより履歴の完全性が保たれる。

### 5.4 I-9（Gate does not trigger execution）の明文化

```
Gateを通過してもExecutionは発生しない
GateはExecutionの「許可」であり「発動」ではない
```

この非結合性により：

- Registrarがgateを通過させても、実行はまだ起きていない
- 実行には別途明示的な指示が必要
- Gateと実行の間に人間の意思決定が介在できる
