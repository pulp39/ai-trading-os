## 3. Core Constraints（Librarian Directive 準拠）

本 Proposal のすべての設計は以下の制約を不変の前提とする。

```
1. production broker path は real_order においてのみ使用する
2. simulated_order パスを real_order に流用しない
3. quantity = 1（固定）
4. attempts = 1（固定）
5. retry 禁止
6. stale 禁止（Freshness 違反は即時中断）
7. dry-run 成功が execution の前提条件
8. execution は before/after 両方の trace_event を必須とする
9. Kill Switch は最優先（すべての判定より前）
10. fallback logic 禁止
```

## 4. Irreversibility Boundary Model

### 4.1 Three Phases

`real_order` は以下の三段階に明確に分割される。

| フェーズ | 名称 | 状態 | 中断可否 |
|---------|------|------|---------|
| Phase A | Pre-Submission | sendorder 送信前・完全中断可能 | **中断可能** |
| Phase B | Submission Boundary | `sendorder` 送信の瞬間（不可逆境界） | **境界点** |
| Phase C | Post-Submission | ブローカー受信後・応答待ち | **中断不可** |

### 4.2 Phase A — Pre-Submission（中断可能）

以下はすべて Phase A に属し、安全に中断できる。

- Gate 通過
- dry-run 通過
- Kill Switch チェック（最終）
- Order payload 構築
- `real_order_ready_to_submit` 記録
- READY context 消費（consumed 遷移）

> **重要**: READY の消費は Phase A の最後、`sendorder` 送信の直前に行う。
> 消費後に送信を中断した場合、当該 READY context は consumed のまま再利用不可となる。

### 4.3 Phase B — Submission Boundary（不可逆境界）

```
sendorder リクエストをブローカーへ送信した瞬間が不可逆境界である。
```

- ブローカーが後に拒否しても、外部への送信試行という事実は変わらない
- したがって「不可逆」は **ブローカー承認時点ではなく、送信時点** から始まる
- `real_order_submitted` 記録は送信直後（Phase B 通過直後）に行う

### 4.4 Phase C — Post-Submission（中断不可）

送信後、ATOS はブローカーの応答を観察するが、送信を「なかったこと」にできない。

- Phase C で観察できる終端状態: `accepted` / `rejected`
- Kill Switch は Phase C でも機能するが、**送信の取消しを意味しない**（Section 8 参照）

## 5. real_order Execution Model

### 4.1 Path 定義

```python
EXECUTION_PATH = "real_order"

# 禁止
# EXECUTION_PATH = "simulated_order"  # 流用禁止
```

`real_order` は以下の接続パラメータを使用する。

| パラメータ | 値 |
|-----------|-----|
| Host | localhost |
| Port | **18080**（本番ポート） |
| Password env | `KABU_API_PASSWORD` |
| Fallback | **禁止** |

### 4.2 Quantity / Attempts 制約

| 制約 | 値 | 変更可否 |
|------|----|--------|
| `quantity` | 1 | Founder Directive 固定・変更不可 |
| `attempts` | 1 | 固定・変更不可 |
| `retry` | 禁止 | |
| タイムアウト | 30秒（P-20260331-003 継承） | |

タイムアウト発生時は `execution_failed` を記録し、`real_order` を終了する。
再試行は行わず、新たな Observation → Preview サイクルを必要とする。

### 5.3 READY Context Consumption Timing

READY context の消費タイミングは以下のように確定する。

```
消費タイミング: sendorder 送信の直前（Phase A 最後・Phase B 進入直前）
```

**消費シーケンス（確定）:**

```
1. Gate = PASS
2. dry-run = PASS
3. 最終 Kill Switch チェック = PASS
4. trace_event: real_order_ready_to_submit
5. READY context → consumed（execution_consumed 記録）
6. sendorder 送信（Phase B 進入）
7. trace_event: real_order_submitted
```

**この順序の根拠:**

消費を送信後に行うと、送信成功・消費失敗という半端状態が生じ、
Duplicate Prevention が機能しない可能性がある。
消費を送信前に行うことで、「送信試行は1回のみ」が制度的に保証される。

### 5.4 Stale 禁止

`real_order` 実行直前に Freshness を再チェックする。

```
if (now - observation_ts) > 60s:
    trace_event: execution_gate_rejected (reason: stale:freshness_expired)
    ABORT
```

これは P-20260330-001（Hard Limits）の freshness > 60s → DATA_INVALID と整合する。

## 5. Dry-Run Requirement

### 5.1 定義

```
dry-run = real_order と同一の入力・判定経路を通るが、
          ブローカーへの実際の発注命令を送信しないテスト実行。
```

### 5.2 実施条件

- `real_order` 着手前に、対象 symbol で dry-run を少なくとも1回実施する
- dry-run は `execution_path = real_order_dry_run` として trace_event に記録する
- dry-run が失敗した場合、`real_order` の実行を禁止する

### 5.3 dry-run 成功基準

| チェック項目 | 基準 |
|-------------|------|
| Kill Switch チェック | 通過 |
| READY Context State チェック | active |
| Founder Approval 確認 | 有効（TTL 内） |
| Freshness チェック | 通過（≤ 60s） |
| Hard Limit チェック | 通過 |
| ブローカー接続疎通確認 | OK |
| Order パラメータ構築 | エラーなし |

dry-run ではブローカーへの `sendorder` 命令は送信しない。
疎通確認は token 取得（`token` エンドポイント）までとする。

### 5.4 Dry-Run が保証すること・しないこと

**保証する（validates）:**

| 保証項目 | 詳細 |
|---------|------|
| 認証 | token 取得成功 |
| API 到達性 | ブローカーエンドポイントへの疎通確認 |
| ランタイム準備状態 | env / port / password 設定確認 |
| ブローカーパス利用可能性 | 18080 への接続確立 |

**保証しない（does NOT validate）:**

| 非保証項目 | 理由 |
|-----------|------|
| 注文受付 | sendorder を送信しないため不明 |
| 価格の有効性 | 市場状況は dry-run 後に変化しうる |
| 数量の有効性 | ブローカー側の検証は sendorder 時のみ |
| 流動性 | リアルタイムの板状況は保証外 |
| 約定確率 | dry-run の対象外 |

```
dry-run = 接続性・認証・ランタイム準備の確認であり、
          注文が受理される保証ではない。
```

### 5.5 dry-run trace_event

```json
{
  "event_type": "real_order_dry_run_passed",
  "ready_context_id": "<対応する ready_context_id>",
  "result": "success | failed",
  "execution_path": "real_order_dry_run",
  "validates": ["authentication", "api_reachability", "runtime_readiness", "broker_path"],
  "timestamp": "<ISO8601>"
}
```

## 6. Traceability Requirements

### 6.1 Terminal State 原則

```
すべての real_order 試行は、必ず以下のいずれか1つの終端状態で終了しなければならない。
サイレント失敗（終端状態なき終了）は禁止する。
```

| 終端状態 | event_type | 適用フェーズ |
|---------|-----------|------------|
| 安全中断 | `real_order_aborted` | Phase A のみ（不可逆境界前） |
| ブローカー拒否 | `real_order_rejected` | Phase C（送信後・ブローカー拒否） |
| ブローカー受付 | `real_order_accepted` | Phase C（送信後・ブローカー受付） |

**各終端状態の定義:**

`real_order_aborted`（Phase A 内のみ使用）
- Kill Switch 発動
- Stale 検出
- dry-run 失敗
- READY context 無効
- payload 構築エラー
→ 外部送信は行われていない。安全な中断。

`real_order_rejected`（Phase B 通過後）
- 送信試行は行われた
- ブローカーがその注文を拒否した
→ 外部送信の事実は残る。

`real_order_accepted`（Phase B 通過後）
- 送信試行は行われた
- ブローカーが注文を受け付けた
→ `accepted` は約定を意味しない。ブローカーが注文リクエストを受理した事実のみを示す。

### 6.2 必須 trace_event シーケンス

**成功パス（accepted）:**

```
1. execution_readiness_evaluated（readiness_state = READY）
2. real_order_dry_run_passed
3. execution_gate_approved
4. real_order_ready_to_submit
5. execution_consumed（READY → consumed）
6. real_order_submitted（Phase B 進入）
7. real_order_accepted               ← 終端
8. execution_performed
```

**拒否パス（rejected）:**

```
1. execution_readiness_evaluated
2. real_order_dry_run_passed
3. execution_gate_approved
4. real_order_ready_to_submit
5. execution_consumed
6. real_order_submitted
7. real_order_rejected               ← 終端
```

**安全中断パス（aborted・Phase A）:**

```
1. execution_readiness_evaluated
2. [dry-run 実施、または Gate チェック中に中断条件検出]
3. real_order_aborted                ← 終端
   （reason: kill_switch:active / stale:freshness_expired / dry_run:failed 等）
```

### 6.3 新設 trace_event 一覧

| event_type | 発火タイミング | Phase |
|-----------|--------------|-------|
| `real_order_dry_run_passed` | dry-run 成功時 | A |
| `real_order_ready_to_submit` | READY 消費直前 | A（最後） |
| `real_order_submitted` | sendorder 送信直後 | B 通過直後 |
| `real_order_accepted` | ブローカー受付確認後 | C（終端） |
| `real_order_rejected` | ブローカー拒否確認後 | C（終端） |
| `real_order_aborted` | Phase A 内の中断時 | A（終端） |

## 7. Kill Switch Integration

### 7.1 チェックポイント

Kill Switch（`KILL_SWITCH.flag`）は以下の三点でチェックする。

| タイミング | Phase | 発動時の終端状態 |
|-----------|-------|----------------|
| Gate 評価開始時（最優先） | A | `real_order_aborted` |
| dry-run 開始時 | A | `real_order_aborted` |
| `real_order_ready_to_submit` 直前（最終） | A | `real_order_aborted` |

### 7.2 Phase A における Kill Switch

Phase A（不可逆境界前）では Kill Switch は送信を完全に防止できる。

```
KILL_SWITCH.flag 存在 → real_order_aborted（reason: kill_switch:active）→ ABORT
```

### 7.3 Phase C における Kill Switch

**Kill Switch は Phase B（送信）通過後の送信取消しを意味しない。**

Phase C で Kill Switch が発動した場合：

- さらなる ATOS アクションを停止する
- retry / 追加送信を阻止する
- モニタリング・アラートをトリガーする
- READY context の復元は行わない

Kill Switch は Phase C において「なかったことにする」機能を持たない。
これは制度上の限界として明示的に受け入れる。

### 7.4 Kill Switch 権限

| 権限 | 主体 |
|------|------|
| KILL_SWITCH.flag 作成（発動） | Founder（手動） / System（自動検出） |
| KILL_SWITCH.flag 削除（解除） | Founder のみ |

## 8. Submission Idempotency Guard

### 8.1 外部重複送信リスク

P-20260401-001（Duplicate Prevention）は内部的な READY context の再利用を防止する。
しかし `real_order` には以下の外部リスクが存在する。

```
プロセス割り込み / ランタイム異常により、
execution_consumed 記録後・real_order_submitted 記録前の
中間状態で処理が中断される可能性がある。
```

この状態で処理が再開された場合、内部的には consumed（再利用不可）であるにもかかわらず、
sendorder が未送信と誤判定される危険がある。

### 8.2 Submission Guard ルール

```
1つの READY context に対して、sendorder 送信は最大1回とする。
これはポリシー（retry禁止）に加え、制度的に保証されなければならない。
```

**実装ルール:**

| 条件 | 処理 |
|------|------|
| 対象 `ready_context_id` の `real_order_submitted` が trace_event に存在する | 送信禁止・`real_order_aborted（reason: duplicate:already_submitted）` |
| `execution_consumed` は記録されているが `real_order_submitted` が存在しない | 送信禁止・`real_order_aborted（reason: duplicate:consumed_without_submission）` |

**Submission Marker:**

`real_order_submitted` イベントが、当該 `ready_context_id` に対する Submission Marker として機能する。

```
if trace_event(event_type='real_order_submitted', ready_context_id=X) exists:
    → 送信禁止（いかなる条件下でも）
    → real_order_aborted（reason: duplicate:already_submitted）
```

### 8.3 consumed / submitted の整合規則

| `execution_consumed` | `real_order_submitted` | 解釈 | 対応 |
|---------------------|----------------------|------|------|
| なし | なし | 正常（未実行） | 実行可能 |
| あり | あり | 正常（送信済み） | 再送信禁止 |
| あり | なし | 異常（中間状態） | 再送信禁止・要調査 |
| なし | あり | 異常（制度違反） | 即時停止・要調査 |

「consumed あり / submitted なし」は中間状態として凍結し、新たな Observation サイクルを要する。

### 8.4 外部べき等性の原則

```
内部 Duplicate Prevention（P-20260401-001）は内部的再利用を防ぐ。
Submission Idempotency Guard は外部送信の一意性を保証する。
両者は独立した防御層として機能し、相互に補完する。
```

## 9. No Fallback Principle

```
real_order において fallback logic は存在しない。

- ブローカー接続失敗時: ABORT（retry なし）
- タイムアウト時: ABORT（retry なし）
- パスワード未設定時: ABORT（代替 env 参照なし）
- ポート変更不可: 18080 固定（fallback なし）
```

## 10. Execution Flow（統合・確定版）

```
━━━━━━━━━━━━━ Phase A（Pre-Submission / 中断可能）━━━━━━━━━━━━━

[1] Kill Switch チェック
    └─ KILL_SWITCH.flag 存在 → real_order_aborted（kill_switch:active）→ END

[2] READY Context チェック
    └─ state != active → real_order_aborted（duplicate:consumed 等）→ END

[3] Founder Approval 確認
    └─ TTL 超過 → real_order_aborted（approval:expired）→ END

[4] Freshness チェック
    └─ > 60s → real_order_aborted（stale:freshness_expired）→ END

[5] Hard Limit チェック
    └─ 違反 → real_order_aborted（hard_limit:violated）→ END

[6] dry-run 実行
    └─ 失敗 → real_order_aborted（dry_run:failed）→ END
    └─ 成功 → real_order_dry_run_passed 記録

[7] execution_gate_approved 記録

[8] real_order_ready_to_submit 記録

[9] Kill Switch 最終チェック（Phase B 進入直前）
    └─ KILL_SWITCH.flag 存在 → real_order_aborted（kill_switch:active）→ END

[10] READY context → consumed（execution_consumed 記録）

━━━━━━━━━━━━━ Phase B（Submission Boundary・不可逆境界）━━━━━━━━━━

[11] sendorder 送信（port 18080）← ここから不可逆

[12] real_order_submitted 記録（送信直後）

━━━━━━━━━━━━━ Phase C（Post-Submission・中断不可）━━━━━━━━━━━━━

[13] ブローカー応答待ち（30秒タイムアウト）
     ├─ 受付 → real_order_accepted 記録（終端）→ [14]
     ├─ 拒否 → real_order_rejected 記録（終端）→ END
     └─ タイムアウト → real_order_rejected 記録（reason: timeout:30s）→ END

[14] execution_performed 記録
```

## 11. Path Isolation（再確認）

本 Proposal により、ATOS の Execution Path 三層が完全に分離される。

```
internal_simulation:
  - ブローカー依存: なし

  - ブローカー依存: 発注
  - KABU_API_PASSWORD: 必須（fallback なし）
  - port: 18080（固定）
  - simulated_order パスからの継承: 禁止
```
