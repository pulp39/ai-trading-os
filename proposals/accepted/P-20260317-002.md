Background
P-20260317-001の実装によりpublic.board_snapshotsへの構造化データ保存が実現しました。現在の観測パイプラインは「1観測 = 1 trace_event」という粒度で記録されています。本Proposalはこれを「1実行 = 1制度単位」として昇格させ、ATOSが「実行主体を持つ制度」へ移行するための基盤を確立します。

Core Distinction（制度的定義）
market_observation = 個別の観測ファクト（1銘柄1時点）
collector_run      = 観測実行の制度単位（1実行N銘柄）
collector_runを「Unit of Observation Execution」として正式定義します。これはconstitution Article 3（Institutional Objects）およびArticle 5（Role Boundary Principle）と完全に整合します。

Change 1 — collector_run event_type の正式定義
collector_runをresearch.trace_eventの正式なevent_typeとして定義します。
最低metadata：
json{
  "event_type": "collector_run",
  "actor": "OpenClaw",
  "scope": "board_snapshot_insert",
  "symbols": ["7203", "8306", "6758"],
  "snapshot_count": 3,
  "status": "success",
  "started_at": "2026-03-17T09:00:00+09:00",
  "completed_at": "2026-03-17T09:00:05+09:00",
  "aab_bundle_id": null
}
```

---

### Change 2 — status の制度定義

`collector_run.status`は以下の値のみを取ります：
```
status ∈ {"success", "partial_failure", "failure"}

success         = 全銘柄のsnapshot INSERT成功
partial_failure = 一部銘柄の失敗（snapshot_countに反映）
failure         = 全銘柄失敗またはCollector起動失敗
```

将来の銘柄ごと失敗への対応およびAAB導入時の整合性を確保するため、この三値定義を制度として固定します。

---

### Change 3 — snapshot_count の定義
```
snapshot_count = successfully inserted snapshots（成功件数のみ）
```

失敗分は`snapshot_count`に含めません。`partial_failure`時は`status`と`snapshot_count`の組み合わせで状況を表現します。

---

### Change 4 — market_observation の位置づけ
```
collector_run      = primary（必須記録）
market_observation = auxiliary（一時的補完、後方互換）
```

Phase 2では`market_observation`を補完として残します。理由は既存trace_eventの連続性保持・デバッグ価値・OpenClaw教育段階での粒度確保です。将来的な廃止は別Proposalで判断します。

---

### Change 5 — AABへの拡張前提

`aab_bundle_id`フィールドを現段階では`null`で予約します。`collector_run`安定後、`aab_bundle_id`を有効化するだけでAABへ移行できます。
```
collector_run（安定）→ aab_bundle_id有効化 → AAB導入

Founder提示文

P-20260317-002 defines collector_run as the institutional unit of observation execution, separating execution from observation facts. It establishes the foundation for AAB without introducing new tables and formalizes operational Git discipline as part of execution governance.
