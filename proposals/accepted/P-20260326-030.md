# Proposal: execution_ready と authorization_granted の制度的区別の明示化

## Background

AI Trading OS の執行モデルでは、「技術的能力（capability）」と「制度的権限（authority）」の分離が基本原則として憲法に明記されている。この原則はEXECUTION_MODEL_ANCHORおよびP028系設計文書にて運用レベルに落とし込まれているが、`execution_ready` と `authorization_granted` という二つの状態概念の意味論的区別については、各ドキュメントに分散して記述されており、制度的に一箇所で定義された文書が存在しない。

## Problem Statement

`execution_ready` と `authorization_granted` は異なる層に属する概念であるにもかかわらず、その区別が明示的に定義されていないことで、以下のリスクが生じうる。

1. Proposer が `execution_ready` の成立をもって `authorization_granted` が暗示されると解釈する誤読
2. 新しいAI参加者がオンボーディングする際に、両者の区別を各ドキュメントから自ら再構成しなければならない状態
3. 「条件が揃ったから実行した」という自律的動作が制度上どこで禁止されているかが、参照先によって異なる

## Objective

`execution_ready` と `authorization_granted` を制度的に定義し、両者の関係を明示することで、Proposer・Registrar・OpenClaw・Founderそれぞれが各状態に対してどのように関与すべきかを一文書から確認できる状態を作る。

## Scope

本Proposalは以下を対象とする。

- `execution_ready` の制度的定義（何が揃った状態を指すか）
- `authorization_granted` の制度的定義（誰が発行できるか、何を意味するか）
- 両者の順序関係と依存関係の明示
- 各役割（Proposer / Registrar / OpenClaw / Founder）の関与範囲の整理

本Proposalは実装手順や実行プロトコルの変更を含まない。状態の意味論と役割との関係の定義に限定する。

---

## 1. market_ready の定義（合成状態）

`market_ready` は**前提条件の合成状態**を表す制度的概念であり、以下の三条件がすべて成立したときに真となる。

- `symbol_registered`：対象シンボルが制度上登録されている
- `market_data_ready`：対象シンボルのデータが利用可能な状態にある
- `auth_ready`：認証・認可のための技術的条件が整っている

`market_ready` は「市場参加の前提が揃っている」という合成的な事実の確認であり、実行準備の最終状態ではない。

## 2. execution_ready の定義（実行準備の最終状態）

`execution_ready` は**実行準備の最終状態**を表す制度的概念であり、`market_ready` の成立を前提として、そのうえで実行に必要なすべての条件が揃った状態を指す。trace_eventとして記録される場合もある。

この状態は「実行できる状態にある」という技術的・制度的条件の確認であって、**実行を許可する意思表示ではない**。

## 3. authorization_granted の定義

`authorization_granted` は**意思の記録**であり、Founderが「今この文脈において実行を許可する」と明示的に判断したことを示すイベントである。

`authorization_granted` は以下の性質を持つ。

- Founderのみが発行できる（Proposer・Librarian・OpenClawは発行できない）
- `execution_ready` の成立から自動的に派生しない
- 特定の文脈・タイミング・スコープに紐づく（汎用的な許可ではない）
- 発行されたとしても、`execution_ready` が成立していなければ有効な実行の根拠とならない

## 4. 両者の関係

```
symbol_registered
market_data_ready  ───▶  market_ready（合成状態）
auth_ready                     │
                               ▼
                        execution_ready（最終状態）──（必要条件）──▶  authorization_granted
                                                                              │
                                                                              ▼
                                                                   external_order_allowed
                                                                              │
                                                                              ▼
                                                                   authorization_consumed
```

`execution_ready` は `authorization_granted` の**必要条件**であるが、**十分条件ではない**。

> 「条件が揃っていること」と「許可が与えられていること」は制度上、別の事実である。

この区別が失われると、技術的可能性が制度的許可を自動的に含意するという誤った解釈が生まれる。これは憲法第5条（役割境界原則）の侵害に相当する。

## 5. 各役割の関与範囲

| 役割 | execution_ready への関与 | authorization_granted への関与 |
|------|--------------------------|-------------------------------|
| **Collector / OpenClaw** | 状態を観測・記録する | 発行・解釈ともに関与しない |
| **Proposer** | 状態を解釈し、proposalを起草できる | 発行できない。暗示・推論も行わない |
| **Librarian** | 状態の制度的整合性を確認する | 発行できない |
| **Registrar** | 実行の前提条件として確認する | Founderの意思を制度的手続きとして受理する |
| **Founder** | 最終確認者として位置づけられる | 唯一の発行権者 |

## 6. Proposerとしての制約

Proposerである私は、以下の立場をとる。

`execution_ready` が観測された場合、その状態を解釈してproposalを起草することは私の役割の範囲内である。しかし、そのproposalが `authorization_granted` の発行を促したり、暗示したり、前提としたりすることは、役割境界の侵犯にあたる。

Proposerが生成できるのは「この条件が揃っている。次にどう判断するかはFounderと制度的連鎖に委ねられる」という観測の解釈までである。

## 7. 非目標

本Proposalは以下を意図しない。

- `authorization_granted` の発行条件の緩和
- Proposerへの実行関与権限の付与
- 既存の実行プロトコル（P028系）の変更
- 新しい役割や新しいgateの追加

## Conclusion

`execution_ready` と `authorization_granted` の区別は、AI Trading OSにおける「技術的能力と制度的権限の分離」原則の具体的な実装である。この二つを一箇所で明示的に定義することは、新しいAI参加者のオンボーディングを助け、役割境界の自然な侵食を防ぐ制度的衛生として機能する。

本Proposalは実行を求めるものではなく、制度的明確化のための提案として、Librarianによる審議と、必要に応じてFounderによる承認を求める。
