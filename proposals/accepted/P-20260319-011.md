Scope: Messager AiiD定義・AAB配送方法・認証トークン概念

Background
P-20260319-010でAABが定義されました。しかし：
AABは存在するが
「どうやってAEEAに届くか」が未定義
本Proposalはこの配送レイヤーを定義します。

核心原則
「信頼は内部で生成され、外部で検証される」

Rule 1: AAB単体では何も起こらない
Rule 2: MessagerがAABトークンを運ぶ
Rule 3: 外部AIは「トークンが正しければのみ」実行する

将来像
Strict ATOS
  ↓（AAB発行）
AAB（信頼トークン）
  ↓（署名付き配送）
Messager AiiD（Strict ATOS内部）
  ↓（正統トークン配送）
AEEA / External Trade Systems（複数・各証券口座）
  Receiver（受信・検証）
  ↓（全検証クリア）
Execution
  ↓
Recorder（DB記録）

Change 1 — Messager AiiDの定義
Messager AiiD：
  役割：AABトークンの署名付き配送主体
  性質：Strict ATOSに属する（内部AiiD）
  責任：トークンの完全性・到達保証
  権限：配送のみ（実行しない・意思決定しない）
なぜ内部に置くか：
もし外部に置くと：
  AAB → 外部Messager → 外部Executor
  → 信頼の境界が外に漏れる

内部に置くと：
  Strict ATOS → Messager → 外部
  → 信頼の起点は常にATOS内部
Messagerは「外部に手を伸ばす唯一の内部AiiD」です。

Change 2 — ATOS Trusted Message Protocol（TMP）
メッセージ構造：
json{
  "tmp_version": "1.0",
  "message_id": "TMP-20260319-001",
  "issued_by": "Messager",
  "aab_id": "AAB-20260319-001",
  "aab_bundle_id": "AAB-20260319-001",
  "target_agent_id": "openclaw",
  "payload": {
    "action_type": "market_observation",
    "symbol": "8306",
    "conditions": {
      "market_hours_only": true,
      "max_executions": 1
    }
  },
  "trust_token": "<署名ハッシュ>",
  "issued_at": "2026-03-19T10:00:00+09:00",
  "valid_until": "2026-03-19T15:30:00+09:00"
}
```

**trust_tokenの定義（修正①反映）：**
```
trust_tokenはAABの完全性と発行主体を証明する識別子である。
trust_tokenは再現可能である必要はないが、検証可能でなければならない。
```

---

### Change 3 — 受信側検証ルール（AEEA側）

外部AIインスタンスは以下を順番に検証します：
```
① trust_tokenの有効性確認
② aab_idがATOSに存在するか確認
③ valid_until が現在時刻より未来か確認
④ target_agent_id が自分自身か確認
⑤ 全条件クリア → 実行
⑥ 1つでも失敗 → 拒否・記録
```

---

### Change 4 — OpenClawの役割暫定分解（修正②反映）
```
OpenClaw（AEEA）の役割：
  ① Receiver  : TMPトークン受信・検証（Messengerとは分離）
  ② Collector : 市場データ取得（現行）
  ③ Recorder  : DB記録（現行）
  ④ Executor  : 実行（将来）
```

**送信と受信の分離：**
```
Messager（ATOS内） → 送信
Receiver（外部）   → 受信・検証
= 絶対に分離する（セキュリティ基本原則）
```

**注意：** 正式定義はP-20260319-012（AEEA Role Decomposition）で行います。
```
