# P-20260322-029
## Human-Operator Real Order Execution Protocol

---

## Background

P-028 Phase 5 established a complete bounded execution framework for a minimal live real_order test.

The system successfully reached:

- execution readiness (`READY_FOR_EXECUTION`)
- validated AAB/TMP alignment
- bounded execution safety guarantees

However, trace_event 294 recorded that OpenClaw declined to perform a live external order submission.

This behavior confirms that:

> Execution capability does not imply autonomous execution authority.

---

## Problem Statement

The system currently lacks a formalized protocol for:

> Human-operator initiated real market execution

Without this, live execution remains:

- possible but undefined
- safe but not institutionally formalized
- authorized but not procedurally governed

---

## Objective

Define a formal protocol for:

> Founder-executed, bounded live real_order transmission

ensuring:

- explicit authorization
- deterministic execution steps
- immediate safety fallback
- full traceability

---

## Scope

This proposal defines:

1. Execution authority model (human vs AI)
2. authorization_consumed trigger definition
3. Founder execution procedure (command-level)
4. Immediate rollback / cancel protocol
5. Post-execution state enforcement
6. Boundary clarification for OpenClaw

---

## 1. Execution Authority Model

Execution roles are defined as:

### OpenClaw (Auxiliary AiiD)
- can validate
- can simulate
- can prepare execution
- can execute bounded Registrar tasks
- **cannot perform live external order submission**
- **cannot trigger authorization_consumed**
- **cannot deploy real capital**

### Founder (Human Operator)
- **is the sole entity authorized to perform live external order submission**
- **cannot delegate this authority to any AI participant**
- holds final execution authority
- bears full accountability for authorization consumption

---

## 2. authorization_consumed Definition

authorization_consumed is triggered:

> At the exact moment a real order payload is transmitted to an external trading API.

Not:

- at order creation
- at validation
- at internal processing

Only:

- at external transmission boundary

authorization_consumed is triggered regardless of:

- transmission success or failure
- order acknowledgment
- order rejection
- transmission_unknown state

Once triggered, it is irreversible.

---

## 3. Founder Execution Procedure

The execution must follow a deterministic sequence:

### Step 1 — Pre-check

- confirm AAB/TMP alignment
- confirm system state = `external_order_allowed`
- confirm authorization not consumed
- confirm execution window is valid

---

### Step 2 — Manual Execution

Founder executes the order using:

- explicitly defined command
- bounded parameters from AAB/TMP

No deviation allowed.

---

### Step 3 — Authorization Consumption

If transmission occurs:

- authorization_consumed = true
- no second execution allowed

---

### Step 4 — Status Observation

Immediately verify:

- order submission success/failure
- current order state

---

### Step 5 — Cancel Protocol

If order is active:

- send cancel request
- confirm cancel result

---

### Step 6 — Safety Lock

System must transition to:

- post_submit_safety_lock
- execution_blocked

post_submit_safety_lock must be applied automatically by the execution script rather than relying on manual human action.

---

## 4. Immediate Rollback Protocol

The system must guarantee:

- cancel path availability
- bounded cancel timeout
- no retry on failure
- no new order during uncertainty

---

## 5. Post-Execution State

After execution:

- system returns to blocked state
- authorization cannot be reused
- no automatic follow-up execution
- trace_event must be recorded

---

## 6. Boundary Definition (OpenClaw)

The following boundary is formally established:

> OpenClaw does not perform live external order submission.

OpenClaw capabilities:

- execution validation
- registrar task execution
- dry-run execution
- bounded simulation
- READY_FOR_EXECUTION state validation

OpenClaw limitations:

- no autonomous external API execution
- no authorization consumption
- no real capital deployment
- no live external order submission

Real market interaction requires explicit Founder execution.

### Proposed BOUNDARY.md addition

OpenClaw (Auxiliary AiiD):

- Can execute bounded Registrar tasks
- Can simulate and validate execution paths
- Can reach READY_FOR_EXECUTION state
- Will NOT autonomously submit live external orders
- Will NOT trigger authorization_consumed
- Real market interaction requires explicit Founder execution

---

## 7. Safety Invariants

The following must always hold:

- one authorization = one transmission
- no retry after transmission
- no execution during unknown state
- no capital escalation
- human accountability at execution boundary
- Founder execution cannot be pre-authorized for future automatic triggering
- post_submit_safety_lock cannot be lifted without new Founder authorization

---

## 8. Non-Goals

This proposal does not define:

- trading strategy
- profit optimization
- multi-order execution
- automation of live trading

---

## Conclusion

P-028 proved that:

> AI can prepare execution, but must not autonomously execute real market actions.

P-029 establishes:

> A formal human-in-the-loop execution protocol that preserves safety, traceability, and institutional integrity.
