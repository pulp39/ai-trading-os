## 1. Background

P-20260328-017（Preview Layer）の実地テスト（2026-03-30）において、Execution Readiness Checkが実市場データの前で正常に機能することが確認された。

しかし、本テストで使用した Soft Limits（スリッページ > 0.3% → REPRICE_REQUIRED 等）は「テスト用ローカル変数」として扱われ、恒常的な制度境界として定義されていなかった。

また、最も重大な判定結果（NOT_READY / DATA_INVALID）の上限境界、すなわち **「絶対に実行してはならない条件」** が制度上未定義であり、Librarianの暗黙的監視によって補完される状態であった。

> これが本Proposalの発端である。

---

## 2. Objective

```
Execution前の「絶対停止条件」を Founder Directive として制度化する
```

Hard Limits は以下の性質を持つ：

- **不可侵領域**（Soft Limitsによる上書き不可）
- **Founder専権**（Proposal levelでの変更不可）
- **例外なし停止**（条件違反時は即 NOT_READY または DATA_INVALID）

---

## 3. Definition

### 3.1 Hard Limits 確定値

| 項目 | 閾値 | 判定結果 | 備考 |
|---|---|---|---|
| データ鮮度 | > 60秒 | DATA_INVALID | 暫定値。将来5〜10秒へ引き下げ予定 |
| スリッページ | > 1.0% | NOT_READY | 絶対停止線（Soft上位と同値だがHardとして確定） |
| 最低流動性（ask_qty） | < 100株 | NOT_READY | 実質約定不能領域 |

### 3.2 Hard と Soft の関係

```
Hard Limits = 不可侵領域（Founder Directive）
Soft Limits = 調整可能領域（Proposal / test単位で設定可変）

Hard > Soft（優先順位絶対）
```

---

## 4. Execution Readiness Check — 判定フロー（確定形）

```python
def check_execution_readiness(observation, order_price, soft_limits):
    # Step 1: Hard Limits（優先・例外なし）
    if observation.freshness_seconds > 60:
        return "DATA_INVALID"

    slippage = abs(order_price - observation.best_ask) / observation.best_ask

    if slippage > 0.010:  # 1.0%
        return "NOT_READY"

    if observation.ask_qty < 100:
        return "NOT_READY"

    # Step 2: Soft Limits（Proposal/test単位で設定）
    if slippage > soft_limits.reprice_threshold:
        return "REPRICE_REQUIRED"

    if observation.ask_qty < soft_limits.liquidity_threshold:
        return "REPRICE_REQUIRED"

    # Step 3: 全条件クリア
    return "READY"
```

## 5. Governance（変更権限）

```
Hard Limits は Founder Directive として制定される。

- 変更には Founder の明示的指示が必要
- Proposal レベルでの上書きは不可
- テスト用パラメータ（Soft Limits）による代替は不可
- 本条項を変更するには新たな Founder Directive が必要
```
