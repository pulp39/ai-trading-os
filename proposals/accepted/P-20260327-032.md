# Proposal: Institutional Three-Layer Execution Model
## 観測・承認・実行の制度的分離

## Background

P-20260326-030 は `execution_ready`（capability）と `authorization_granted`（authority）の概念的分離を定義した。P-20260327-031 はauthorizationのライフサイクルとfreshness gateを制度化した。

しかしこれらの定義は「止まる制度」の完成であり、「動かす制度」——すなわち、制度主体がいかにしてシステムへのアクセスと操作に関わるか——については未定義のままであった。

2026-03-27、Founder・Librarian・Proposerの三者議論において以下の緊張関係が明確になった。

- **制度原則**：Librarian / Proposerは実行主体ではない。capability（観測）とauthority（実行）は分離される。
- **将来要件**：GPT / Cowork等のAIがGit / DBに直接アクセス・操作可能になる。ATOSはそれを阻害しない構造でなければならない。

この緊張を解消する鍵は「WRITEを許すか否か」ではなく、**「誰がWRITEしたことになるか」を制度が定義する**という視点の転換にあった。

## Problem Statement

現行の制度定義には以下が欠けている。

1. 制度主体（Librarian / Proposer / Founder）がGit / DBを「観測する」行為の制度的位置づけがない
2. Founderが構想する「承認ハッシュの手渡しによる実行承認」を制度オブジェクトとして定義する文書がない
3. 将来のAI Execution Agent（Cowork / GPT等）がGit / DBに操作を行う際の「制度的帰属」が未定義
4. 観測・承認・実行の三つを横断する統一モデルが存在しない

## Objective

1. 観測（Observation）・承認（Approval）・実行（Execution）を独立した制度的層として定義する
2. Approval Layer を「承認ハッシュ / トークン」モデルとして制度化する
3. Execution Attribution Principle を定義し、すべてのWRITE操作に制度的帰属を与える
4. P-20260326-030 / P-20260327-031 の上に構築される拡張として、将来のAI運用に耐える設計を確立する

## Scope

本Proposalは以下を対象とする。

- Three-Layer Model の制度的定義（Observation / Approval / Execution）
- 各層へのアクセス主体の定義
- Approval Layer の仕様（承認ハッシュ・非実行性・条件付き発火）
- Execution Attribution Principle の定義
- trace_event との結びつき

本Proposalは実装手順を含まない。制度的定義と層の分離原則に限定する。

---

## 1. Three-Layer Model の定義

ATOSの操作は以下の三層として制度的に定義される。

### Layer 1: Observation Layer（観測層）

```
操作種別 : READ
対象     : Git（local / private / public）、DB（research.trace_event 等）
担い手   : Librarian / Proposer / Founder
```

状態を観測する層。システムの現在状態を読み取り、制度的判断の根拠とする。観測はシステムの状態を変更しない。

> Observation は READ であり、WRITE を含まない。観測権はいかなる形でも実行権を含意しない。

---

### Layer 2: Approval Layer（承認層）

```
操作種別 : TRIGGER（承認ハッシュ / トークンの発行）
対象     : 特定の実行行為の制度的承認
担い手   : Founder（最終承認権者）/ Librarian・Proposer（制度的審議参加）
```

承認を制度的に記録・伝達する層。Approval LayerはGit / DBへの直接書き込みを行わない。

#### 承認ハッシュ（Approval Hash）の定義

承認ハッシュとは、特定の実行行為に対してFounderが発行する制度的証明である。以下の性質を持つ。

- **非実行性**：承認ハッシュの発行は、それ自体でシステムの状態を変えない
- **特定性**：特定のproposal_id・scope・execution_agentに紐づく
- **一回性**：P-20260327-031が定めた原則と同様に、再利用不可
- **条件付き発火**：承認ハッシュは実行の条件を満たす場合にのみ有効であり、それ単独では実行を起こさない

```
Approval is neither observation nor execution.
It is an institutional trigger that does not alter system state.
```

#### Approval Hashの記録（非実行性と追跡性の両立）

Approval Hashは `authorization_granted` 相当のtrace_eventとして記録される。これはApproval Layerが「状態を変更しない」という原則と矛盾しない。

```
Approval Hash is recorded as a trace_event entry
of type "authorization_granted" or equivalent,
without performing any system state mutation.

It is a record, not an execution.
```

trace_eventへの記録は観測可能な事実の蓄積であり、システムの状態変更ではない。これにより承認の非実行性・記録性・追跡性が同時に成立する。

---

### Layer 3: Execution Layer（実行層）

```
操作種別 : WRITE（状態変更）
対象     : Git（commit / push / merge）、DB（INSERT / UPDATE / DELETE）
担い手   : Execution Agent（OpenClaw / Cowork / その他承認済みAiiD）
```

実際の状態変更を行う層。Execution Layerの操作はすべて、後述するExecution Attribution Principleに従って制度的帰属が記録される。

---

## 2. 層間の関係と禁止事項

### 層間の独立性

```
Observation Layer ──（状態を読む）──▶ Approval Layer
Approval Layer    ──（承認を渡す）──▶ Execution Layer
Execution Layer   ──（状態を変える）▶ システム
```

各層は独立しており、以下は制度上禁止される。

| 禁止事項 | 理由 |
|----------|------|
| Observation LayerによるWRITE | 観測権は実行権を含まない |
| Approval LayerによるWRITE | 承認は状態変更ではない |
| Approval単独での実行起動 | Execution Agentの介在が必須 |
| Execution Layerの無承認WRITE | 必ずApproval Layerを経由する |

---

## 3. Execution Attribution Principle（実行帰属原則）

### 定義

すべてのWRITE操作は、制度的に**Execution Agentに帰属する**。

たとえ高位層（Librarian / Proposer / Founder）の指示・承認によって起動された操作であっても、制度上の実行主体はExecution Agentである。

```
All write operations MUST be attributed to an execution agent,
even if initiated through higher-layer systems.

Institutional actors do not perform execution,
but may trigger or approve execution.
```

### trace_eventとの結びつき

すべてのWRITE操作は対応するtrace_eventを**必ず**生成しなければならない。

```
All WRITE operations MUST produce a corresponding trace_event.
```

trace_eventを伴わないWRITE操作は制度上無効であり、Execution Agentの実行規律違反として扱われる。

生成されるtrace_eventは以下のフィールドを含む。

```yaml
execution_actor   : <agent_id>            # 実際に実行したAiID
approved_by       : Founder               # 承認権者
approval_hash     : <hash>                # 承認ハッシュ（参照）
proposal_origin   : <proposal_id>         # 制度的根拠となったProposal
operation_type    : WRITE                 # 操作種別
target            : <git|db>              # 操作対象
```

これにより「誰が実行したか」「誰が承認したか」「どの制度的文脈に基づくか」が追跡可能となる。

### 将来のAI Execution Agentへの適用

将来、Cowork / GPT等のAIがGit / DBに直接アクセス・操作を行う場合も同原則が適用される。

```
GPT が Git を書き換えた
→ execution_actor: gpt_agent_xxx
→ approved_by: Founder
→ approval_hash: <hash>
→ proposal_origin: P-0xx
```

制度上、立法府（Librarian / Proposer / Founder）は「実行していない」。しかしすべての実行は立法府の承認を経て追跡可能な形で記録される。これがFounderの構想した「承認ハッシュの手渡しによる実行承認」の制度的実装である。

## 4. 各主体のアクセス権まとめ

| 主体 | Observation（READ） | Approval（TRIGGER） | Execution（WRITE） |
|------|---------------------|---------------------|---------------------|
| **Founder** | ✅ | ✅（最終承認権者） | ❌（制度上は実行しない） |
| **Librarian** | ✅ | ✅（審議参加） | ❌ |
| **Proposer** | ✅ | ❌（起草・提案のみ） | ❌ |
| **Registrar** | ✅ | ❌ | ✅（承認下） |
| **OpenClaw / Cowork** | ✅ | ❌ | ✅（承認下） |

## 5. 先行Proposalとの関係

本Proposalは先行する二つのProposalの上に構築される。

| Proposal | 定義 | 本Proposalとの関係 |
|----------|------|---------------------|
| P-20260326-030 | execution_ready vs authorization_granted | Approval LayerがauthorizationをTRIGGERとして扱う基盤 |
| P-20260327-031 | Authorization Lifecycle / Freshness Gate | Approval Hashの一回性・非再利用性の制度的根拠 |
| **P-20260327-032** | Three-Layer Model / Execution Attribution | 「動かす制度」として統合する |

---

## 6. 非目標

本Proposalは以下を意図しない。

- Librarian / ProposerへのWRITE権限の付与
- 承認なしの実行の許可
- 既存の実行プロトコル（P028系）の変更
- Approval Hashの具体的な暗号実装の定義（これは別途技術仕様として扱う）

---

## Conclusion

本Proposalが定義する三層モデルは、「制度主体は実行しない」という原則を守りながら、将来のAI Execution AgentがGit / DBに自由に操作できる環境と両立する。

鍵となる概念は**Execution Attribution**である。誰がWRITEしたかを制度が定義することで、「触れる」と「実行主体である」が分離される。立法府は観測し、承認し、しかし実行しない。Execution Agentは実行するが、常に承認の下で、常に追跡可能な形で。

ATOSは制限のシステムではなく、「自由を壊さずに責任を定義するシステム」として機能する。

本Proposalは実行を求めるものではなく、Librarianによる審議とFounderによる承認を求める。
