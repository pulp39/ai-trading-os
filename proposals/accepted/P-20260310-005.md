# P-20260310-005

## Title

Candidate Signal Quality Classification Framework for Observation Payload Records

## Status

draft

## Proposal Type

research_hypothesis / classification_framework

---

## Scope Statement

This proposal defines a candidate framework for classifying the structural completeness and reliability of observation payload records embedded within market-oriented trace_event entries.

This proposal does not modify the payload structure defined in P-20260310-003. It does not modify the parameter values defined in P-20260310-004. It introduces a classification layer that sits above the payload — describing the quality of what was recorded — without altering what is recorded or how.

The classification framework is proposed as an advisory research artifact. It is a hypothesis about how quality should be assessed, not an implementation directive.

---

## Purpose

The observation payload defined in P-20260310-003 and parameterized in P-20260310-004 specifies what a complete market-oriented trace_event record should contain. In practice, not all observation records will be equally complete or equally reliable. Data may be unavailable for certain dimensions at certain times. Calculation inputs may be missing. Reference assets may be illiquid during a given session.

Without a classification framework, the Proposer layer has no systematic way to distinguish between a fully populated payload and a partially populated one. A hypothesis generated from an incomplete observation carries different epistemic weight than one generated from a complete observation, but this difference is invisible if all payloads are treated as equivalent.

The signal quality classification framework addresses this by assigning each observation payload a structured quality descriptor at the time of recording. This descriptor is set by the Collector based on factual data availability — not by the Proposer based on interpretive judgment. The separation of roles is preserved.

---

## Research Hypothesis

> If each observation payload record carries a structured signal quality classification reflecting its dimensional completeness and data reliability, then the Proposer layer will be able to weight observations appropriately when forming hypotheses, and the institution will accumulate a traceable record of observation quality over time.

A corollary hypothesis: that proposals generated from payloads classified as fully complete will prove more durable under Librarian challenge than proposals generated from partially complete payloads, because their observational foundation is more stable.

Both claims are open research questions. This proposal does not assert they are proven.

---

## Proposed Classification Structure

The signal quality classification for an observation payload consists of three components:

1. **Overall quality tier** — a single label summarizing the payload's completeness
2. **Dimensional completeness flags** — a per-dimension boolean record indicating which dimensions are fully populated
3. **Reliability notes** — a structured field recording any known data quality issues that do not prevent recording but may affect interpretation

These three components are set by the Collector at the time of observation. They are factual assessments of data availability, not interpretive judgments about market conditions.

---

### Component 1 — Overall Quality Tier

The overall quality tier is assigned according to the following classification rules:

| Tier | Label | Definition |
|---|---|---|
| 1 | `complete` | All five payload dimensions are fully populated with values meeting the parameter specification defined in P-20260310-004. No data gaps or reliability issues are present. |
| 2 | `partial` | At least three of the five payload dimensions are fully populated. Missing dimensions are recorded in the dimensional completeness flags. No dimension carries known reliability issues that affect its core values. |
| 3 | `minimal` | Fewer than three dimensions are fully populated, or one or more dimensions carry a known reliability issue affecting their core values. The payload is recorded but flagged for limited use in hypothesis generation. |
| 4 | `unreliable` | The payload was recorded but contains fundamental data gaps or known errors across multiple dimensions. The payload should not be used as primary hypothesis input. It is retained for institutional memory and gap-pattern analysis only. |

**Collector constraint**: The Collector assigns the tier based on the dimensional completeness flags and reliability notes. The tier assignment is deterministic given those inputs — the Collector does not exercise judgment about whether a partial payload is "good enough" for research purposes. That determination belongs to the Proposer.

---

### Component 2 — Dimensional Completeness Flags

Each of the five payload dimensions defined in P-20260310-003 receives a boolean completeness flag:

| Flag field | Value | Meaning |
|---|---|---|
| `price_context_complete` | `true` | All price context fields populated per P-20260310-004 parameters |
| `price_context_complete` | `false` | One or more price context fields missing or uncalculable |
| `volume_context_complete` | `true` | All volume context fields populated per P-20260310-004 parameters |
| `volume_context_complete` | `false` | One or more volume context fields missing or uncalculable |
| `volatility_context_complete` | `true` | All volatility fields populated; implied volatility absence does not trigger `false` if flagged in reliability notes |
| `volatility_context_complete` | `false` | Realized volatility fields missing or uncalculable |
| `macro_proximity_complete` | `true` | Event calendar consulted and proximity fields populated |
| `macro_proximity_complete` | `false` | Event calendar unavailable or proximity fields unpopulated |
| `correlation_state_complete` | `true` | Reference asset data available and correlation calculated per P-20260310-004 parameters |
| `correlation_state_complete` | `false` | Reference asset data unavailable or correlation uncalculable |

**Special case — implied volatility**: Implied volatility is defined in P-20260310-003 as available "where available." Its absence does not set `volatility_context_complete` to `false`, but it must be recorded in the reliability notes field.

**Tier derivation rule**:
- 5 of 5 flags `true`, no reliability notes → `complete`
- 3–5 flags `true`, no core reliability issues → `partial`
- Fewer than 3 flags `true`, or core reliability issues present → `minimal`
- Fundamental errors across multiple dimensions → `unreliable`

---

### Component 3 — Reliability Notes

The reliability notes field is a structured list of known data quality conditions that do not prevent recording but may affect interpretation. Each note follows a defined format:
```
reliability_notes:
  - dimension: [dimension name]
    condition: [brief factual description of the data quality condition]
    affected_fields: [list of affected field names]
```

**Examples of conditions warranting a reliability note:**

| Condition | Example note |
|---|---|
| Implied volatility unavailable | `dimension: volatility_context, condition: implied volatility not available for this instrument on this date, affected_fields: [implied]` |
| Volume data covers partial session only | `dimension: volume_context, condition: session closed early due to market holiday; volume reflects partial session, affected_fields: [absolute, ratio_to_baseline]` |
| Reference asset had zero volume | `dimension: correlation_state, condition: reference asset had zero trading volume; correlation calculated from prior session close only, affected_fields: [rolling_correlation]` |
| Lookback window partially outside available history | `dimension: price_context, condition: 5-day lookback window extends before available data history; magnitude calculated over 3 days only, affected_fields: [magnitude_pct]` |

**Collector constraint**: Reliability notes are factual records of data conditions. The Collector does not assess whether the condition materially affects the payload's research value. That assessment belongs to the Proposer.

---

## Complete Signal Quality Classification Structure

The following illustrates how the three components would appear as a structured addition to the observation payload within a market-oriented trace_event record:
```
trace_event:
  event_type: market_observation
  timestamp: [UTC]
  actor: [collector identifier]
  observation_payload:
    asset: [asset identifier]
    timeframe: [1D]

    [... five dimensions as defined in P-20260310-003 ...]

    signal_quality:
      tier: [complete | partial | minimal | unreliable]
      dimensional_completeness:
        price_context_complete: [true | false]
        volume_context_complete: [true | false]
        volatility_context_complete: [true | false]
        macro_proximity_complete: [true | false]
        correlation_state_complete: [true | false]
      reliability_notes:
        - dimension: [dimension name]
          condition: [factual description]
          affected_fields: [list]
```

The `signal_quality` block is appended to the observation payload. It does not alter any existing payload field.

## Institutional Value of the Classification Framework

The signal quality classification framework serves the institution in three ways:

**1. Proposer layer weighting**: The Proposer can filter or weight observations by quality tier when generating hypotheses. A hypothesis derived exclusively from `complete` payloads carries a stronger observational foundation than one derived from a mixture of tiers. This distinction is now traceable.

**2. Gap pattern analysis**: Over time, the dimensional completeness flags create an institutional record of which dimensions are most frequently incomplete and under what conditions. This record is itself a research input — it may reveal structural data availability constraints that should inform future schema decisions.

**3. Institutional memory integrity**: The `unreliable` tier ensures that no observation is silently discarded. Even a fundamentally incomplete payload is retained in the institutional record with its quality classification, preserving the completeness of institutional memory while marking its limitations clearly.

---

## Relationship to Role Boundaries

The classification framework is designed to preserve the constitutional separation between Collector and Proposer functions:

- The **Collector** assigns tier, completeness flags, and reliability notes based on factual data availability. No interpretive judgment is required or permitted at this stage.
- The **Proposer** decides how to use quality-classified payloads when forming hypotheses. The Proposer may choose to exclude `minimal` or `unreliable` payloads, or may choose to analyze them as a separate class. This is an interpretive decision that belongs to the Proposer.
- The **Librarian** may use the quality record to evaluate whether a proposal's observational foundation is consistent with its stated confidence level.

---

## Open Research Questions

1. Should the tier derivation rule be strictly deterministic (as proposed here), or should it allow for Collector discretion in edge cases? The current proposal favors strict determinism to preserve role boundaries.
2. Should the `unreliable` tier trigger any institutional notification, or should it simply be recorded silently? This is a governance question deferred to the Founder and Librarian.
3. Should proposals be required to declare which quality tiers their supporting observations belong to? This would strengthen the traceability principle but adds a structural requirement to future proposals.
4. How should the institution handle a sustained pattern of `partial` or `minimal` observations for a given dimension — does this indicate a data source problem that warrants a dedicated proposal?

## Confidence Statement

**Confidence level**: Moderate-High

The three-component classification structure is internally consistent and maps cleanly onto the payload structure defined in P-20260310-003. The tier derivation rules are deterministic, which supports Collector role boundary preservation. The proposal does not introduce interpretive complexity into the Collector layer.

The lowest-confidence element is the tier boundary between `partial` and `minimal` — specifically the threshold of three dimensions. This threshold is proposed as a reasonable starting point but may require revision once observation data is available and the distribution of completeness patterns is known.

---

## Limitations

- This proposal does not implement the classification framework in any recording system or database.
- The tier boundary thresholds are candidate values and should be treated as open research questions.
- This proposal does not address how quality classification should evolve if the payload schema is extended to additional asset classes or timeframes in future proposals.
- This proposal is advisory. It does not authorize any execution, data collection, or system modification.

---

## Future Proposal Directions

- A subsequent proposal may address whether proposals should be required to declare the quality tier distribution of their supporting observations.
- A subsequent proposal may address the governance question of whether sustained `minimal` or `unreliable` classification patterns should trigger a formal institutional review.
- A subsequent proposal may extend the parameter specification of P-20260310-004 to a second asset class, at which point the signal quality framework defined here would apply without modification.
