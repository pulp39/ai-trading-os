# P-20260321-027（Rev.1）
## Real Order Authorization Conditions
### (Conditional Authorization Framework)

**Date:** 2026-03-21
**From:** Librarian (GPT)
**Co-designed with:** Proposer (Claude)
**Status:** Rev.1 — Ready for Acceptance Discussion
**Type:** Execution Governance / Authorization Layer
**Constitutional Impact:** None

---

## 1. Background

P-024 established execution_blocked as a valid institutional outcome.
P-025 introduced structured block taxonomy.
P-026 enabled hypothesis generation from blocked events.

From H-20260321-001, the following was clarified:

- B-4 (real_order_not_authorized) is not an error
- It correctly protects an unimplemented capability
- The problem is not restriction, but absence of authorization conditions

---

## 2. Objective

Define institutional conditions under which:

```
real_order_authorized = true
```

can be safely and intentionally set.

This proposal introduces:

- authorization condition set
- readiness validation requirements
- authorization decision process
- integration with execution_gate and execution_blocked

---

## 3. Core Principle

Authorization is not relaxed.
Authorization is earned.

real_order execution must never be:

- implicit
- automatic
- unbounded

---

## 4. Authorization Condition Set

real_order_authorized may be set to true only if ALL conditions are satisfied:

**A-1 Execution Gate Validated**
- execution_gate_passed exists and is verified

**A-2 Preview Integrity Confirmed**
- estimated_price exists and is within acceptable bounds
- preview reflects actual execution conditions

**A-3 Risk Bound Defined**
- max position size is explicitly defined
- loss exposure is bounded
- risk parameters must be explicitly defined in the task artifact
  (not implied or inherited from preview)

**A-4 Environment Safety Confirmed**
- execution environment is controlled (test / sandbox / limited scope)
- no uncontrolled external side-effects

**A-5 Registrar Explicit Authorization**
- authorization must be explicitly granted by Registrar
- authorization is event-based, not state-based

---

## 5. Authorization Model

Authorization is:

- conditional
- explicit
- time-bound
- scope-bound

Example:

```json
{
  "real_order_authorized": true,
  "authorization_scope": "single_execution",
  "authorization_valid_from": "<ISO8601>",
  "authorization_expiry": "immediate_after_execution",
  "authorized_by": "registrar"
}
```

Notes:
- `authorization_valid_from` defines when authorization becomes effective
- ensures traceability of authorization timing
- aligns with gate_validity_window concept

---

## 6. Integration with Existing System

### 6.1 With execution_gate

execution_gate_passed is necessary but not sufficient.

Flow:

```
execution_gate_passed
→ authorization check
→ execution OR block
```

### 6.2 With execution_blocked

If authorization conditions are not met:

```
block.code = B-4
block.reason = real_order_not_authorized
```

Thus B-4 transitions from:

```
"permanent block"
→ "conditional block"
```

> **Note:** B-4 remains a hard block until ALL A-1 ~ A-5 are satisfied.
> Partial satisfaction does not reduce severity.

---

## 7. Authorization Decision Process

```
Step 1: Preconditions validated (A-1 ~ A-5)
Step 2: Registrar evaluates conditions
Step 3: Registrar issues explicit authorization event
Step 4: execution may proceed
```

No implicit transition is allowed.

---

## 8. Non-Scope

This proposal does NOT include:

- real_order execution implementation
- capital allocation
- automatic authorization
- strategy selection

---

## 9. Expected Effect

P-027 enables:

- controlled transition from simulation to execution
- preservation of safety guarantees
- extension of execution capability without breaking invariants

```
P-024 proved the system can stop.

P-026 allowed the system to learn from stopping.
P-027 defines when the system may proceed.

Authorization is not removed.
Authorization is defined.
```
