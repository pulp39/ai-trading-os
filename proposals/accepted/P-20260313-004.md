## 1. Research Hypothesis

Defining an operational integration framework for OpenClaw as the
collector_core occupant on VM-A, with kabuStation as the initial
data source, will transition the Observation Layer from institutional
definition to active sensing capability, enabling evidence-based
proposal generation.

## 3. Supporting Observations

- P-20260313-003 established collector_core as an AiiD institutional
  seat with OpenClaw as occupant on VM-A. The seat exists but no
  operational parameters have been defined.
- The Founder policy directive specifies kabuStation market data as
  the initial and Phase A-exclusive data source. Future expansion
  (news, sentiment, macro) is anticipated but deferred.
- P-20260310-011 defined OpenClaw's permission boundaries through the
  Stage evaluation framework. P-20260313-003 Phase A inherited those

- The two-tier recording model (OpenClaw -> observation report ->
  Registrar -> trace_event) was confirmed in P-20260313-003 section 4.3
  and must be implemented consistently here.

## 4. Proposal Content

### 4.1 Operational Scope (Phase A)

```
occupant:          OpenClaw
seat:              collector_core
deployment:        VM-A
phase:             Phase A
data_source:       kabuStation API (only)
observation_mode:  read-only / passive
```

No data sources other than kabuStation API are permitted during Phase A.
Expansion to additional sources requires a separate proposal.

### 4.2 kabuStation Data Scope

The following data fields are permitted for collection during Phase A:

```
symbol            (ticker / instrument identifier)
price             (current / last trade price)
volume            (trading volume)
previous_close    (prior session closing price)
order_book        (bid/ask depth, if available via API)
timestamp         (ISO8601, server-side)
```

Fields not listed above require a separate proposal to include.

### 4.3 Observation Report Format (Phase A Provisional Schema)

OpenClaw shall generate observation reports in the following format:

```yaml
observation_type: market
source: kabuStation
collector_id: collector_core
phase: Phase A
symbol: <ticker>
timestamp: <ISO8601>
data:
  price: <value>
  volume: <value>
  previous_close: <value>
  change: <value>
  order_book: <bid/ask depth or null>
analysis:
  summary: <optional one-line descriptive summary>
  hypothesis: <optional non-binding hypothesis, clearly flagged>
```

Rules:
- `analysis` fields are optional and non-binding.
- Hypotheses in `analysis.hypothesis` must be clearly flagged as
  non-binding and must not constitute execution instructions.
- The schema is provisional for Phase A. Schema changes require a
  separate proposal.

### 4.4 Observation -> trace_event Flow

The two-tier recording model established in P-20260313-003 is maintained:

```
collector_core (OpenClaw)
      |
      v
observation report (YAML format per 4.3)
      |
      v
Registrar
      |
      v
trace_event (event_type: canonical_observation)
```

OpenClaw does not write directly to the trace_event system.
The Registrar is the sole institutional recorder of observation events.

trace_event structure for canonical_observation:

```json
{
  "event_type": "canonical_observation",
  "agent_id": "collector_core",
  "actor_type": "collector",
  "content": "<summary of observation>",
  "metadata": {
    "observation_class": "canonical",
    "collector_id": "collector_core",
    "source": "kabuStation",
    "symbol": "<ticker>",
    "phase": "Phase A"
  }
}
```

### 4.5 Operational Constraints

The following constraints apply to OpenClaw during Phase A:

```
Permitted:
  - passive market data retrieval via kabuStation API
  - generation of observation reports per section 4.3
  - optional non-binding hypothesis generation (flagged)

Prohibited:
  - order execution of any kind
  - repository modification
  - direct trace_event write
  - proposal enactment
  - access to data sources other than kabuStation
  - retention of data beyond the current observation cycle
    without explicit Registrar recording
```

### 4.6 Observation Frequency

Observation frequency is not mandated by this proposal.
Operational frequency shall be determined by the Founder
and communicated to the Registrar for implementation.
This proposal establishes the framework; scheduling is operational.

### 4.7 Future Expansion Path

Phase A is limited to kabuStation data. Future expansion may include:

```
Phase B (future proposal):
  - news feeds
  - sentiment data
  - macro economic indicators
  - other external signals
```

Each expansion requires a separate proposal approved through the
standard governance process.

### 4.8 Relationship to Existing Framework

This proposal does not modify:
- the Constitution
- Founder authority
- P-20260312-006 (Collector Governance Framework)
- P-20260313-003 (Collector Core Seat Establishment)
- Librarian review authority
- Registrar execution authority

This proposal operationalizes the collector_core seat established
in P-20260313-003 by defining data scope, observation format,
and recording flow for Phase A.

### 4.9 Implementation Requirements

If adopted:
- trace_event: operational_governance_rule_enacted
- OpenClaw configured on VM-A per sections 4.1-4.5
- Registrar briefed on canonical_observation recording flow (4.4)
- Observation frequency determined by Founder (operational)

## 5. Possible Experiments

Upon operationalization, the first research experiment is:

```
Observation target: kabuStation market data (1-3 symbols)
Collection method:  OpenClaw passive retrieval
Output:             canonical_observation trace_events via Registrar
Evaluation:         Does the two-tier recording model function
                    as designed? Are observation reports sufficient
                    for Proposer hypothesis generation?
```

## 6. Confidence Statement

High. This proposal is a direct operationalization of the framework
established in P-20260313-003 and the Founder policy directive.
The two-tier recording model, data scope, and permission constraints
are internally consistent and traceable to prior institutional records.

## 7. Limitations

- OpenClaw's technical configuration on VM-A is an operational matter
  outside this governance proposal.
- Observation frequency is not defined here; Founder determination required.
- The provisional schema (4.3) may require revision based on actual
  kabuStation API output. Schema changes require a separate proposal.
- Phase A is strictly kabuStation-only. Any additional data source
  requires a separate proposal regardless of similarity.
