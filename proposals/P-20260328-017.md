## 1. Background

AI Trading OSは現在、以下の制度フローを持つ。

```
Observation → Dialogue → Proposal → Approval → Execution
```

（五段構成：P-20260328-033により確立）

しかし、`simulated_order`から`real_order`への移行において、

> **実行の安全性を制度的に保証する層が未定義**

である。これが本Proposalの発端である。

---

## 2. Objective

```
simulated_order から real_order への unsafe transition を防ぐ
```

「正確な価格を予測する」ではなく、「この条件なら実行してはいけない」を制度として定義することが目標である。

```
Preview Layer（実行前評価層）
```

## 5. Preview Layer Definition

```
Preview Layerとは、
市場観測と実行意図の整合性を検証し、
Execution Readiness状態を生成する非実行評価層である。
```

### Properties

- READ / WRITE / TRIGGER を発生させない
- 制度的確定を行わない（Founder承認が必要）
- Approval Layerへの入力（Readiness状態）を生成する
- Observation Layerとは明確に別物（ApprovalのInput生成層）

---

## 6. Execution Readiness Check

### 定義

```
Execution Readiness Checkとは、
想定実行条件と市場状態の整合性を検証し、
実行可否を制度的に判定するメカニズムである。
```

### 検証軸

① **価格整合性**（estimated_price vs 市場価格）
② **市場条件**（スリッページ・流動性）
③ **データ整合性**（鮮度・欠損・矛盾）

---

## 7. Readiness States

| 状態 | 意味 |
|------|------|
| `READY` | 実行可能 |
| `NOT_READY` | 実行停止（Hard Limit違反） |
| `REPRICE_REQUIRED` | 再価格算出が必要 |
| `DATA_INVALID` | データ整合性エラー |
| `HUMAN_REVIEW_REQUIRED` | 人間レビューが必要 |

---

## 8. Threshold System（二層構造）

### 8.1 Hard Limits（固定・Founder Directive）

Founder Directiveとして定義される不変条件。

| パラメータ | 説明 |
|-----------|------|
| 最大スリッページ | 超過時は強制 NOT_READY |
| 最低流動性 | 下回った場合は強制 NOT_READY |

### 8.2 Soft Limits（可変・Proposal単位）

Proposal単位で定義可能な戦略パラメータ。

| パラメータ | 説明 |
|-----------|------|
| 許容価格乖離 | 超過時は REPRICE_REQUIRED |
| 再価格条件 | HUMAN_REVIEW_REQUIRED トリガー条件 |

### 判定ロジック

```
if Hard Limit違反:
    NOT_READY（強制停止）

elif Soft Limit違反:
    REPRICE_REQUIRED または HUMAN_REVIEW_REQUIRED

else:
    READY
```

---

## 9. Price Preview / Binding

```
estimated_price を算出し、
条件を満たす場合のみ実行価格として拘束（Bind）可能とする。
```

価格の予測ではなく、**実行価格の制度的確定**が目的である。

---

## 10. Trace Event Design（三点発火）

### 10.1 Preview判定時（最重要）

```
event_type: execution_readiness_checked
```

記録内容：判定状態・判定理由・入力データ（スリッページ値・流動性値等）

目的：なぜ実行が止まったか/通過したかの監査証跡。`NOT_READY`・`REPRICE_REQUIRED`・`READY`の根拠を記録。

### 10.2 Approval時

```
event_type: execution_approved
```

目的：Founderの承認を制度記録として残す（既存パターンと同様の役割）。

### 10.3 Execution後

```
event_type: real_order_executed
```

目的：実際の約定結果の記録。Execution Attributionの三項分解もここで確定。

---

## 11. Safety Principle

```
本層は市場を予測しない。
実行の安全性のみを判定する。
```

**この原則の制度的意義：** Preview Layerが「予測システム」に変質するリスクを制度的に閉じる。将来的に「価格予測機能をPreviewに組み込む」という提案がなされた場合、本Safety Principleが「それはPreview Layerではなく、別のProposalが必要」と制度的に答える。

---

## 12. Impact

- **Execution安全性の制度化**：real_order実行前に必ず安全判定が入る
- **real_order接続の前提確立**：simulated_order → real_order の制度的橋渡し
- **監査可能性の大幅向上**：三点trace_eventにより「判定→承認→実行」が完全に追跡可能
